# NAN 2026 Murder Mystery — Development History

> 기존 `history.md`(Vanguard-Agent)와는 별개 프로젝트이므로 신규 히스토리 파일로 분리한다.
> 형식은 프로토콜 2 (Phase 단위, "무엇을 했는가" + "Technical Rationale")를 그대로 따른다.

## Phase 1-Documentation: 설계 인계 문서화 및 최소 검증 계획 수립 (Completed)

- 사전 대화창에서 확정된 설계(사건 진실, 페르소나 8유형, 증거 14종, 3계층 AI 아키텍처, 점수/UI 스펙)를 정식 문서 4종으로 정리했다: `00_prototype_validation_plan.md`, `01_truth_bible.md`, `02_persona_design.md`, `03_character_sheets.md`, `04_game_loop_flow.json`.
- 캐릭터 시트를 "플레이어 공개용"과 "AI 시스템 프롬프트용"이 하나의 소스에서 `visibility` 필드(`player`/`ai_only`/`both`)로 파생되는 단일 스키마 구조로 설계했다.
- 게임 루프를 flow-to-ai Semantic IR v0.1.0(Top-to-Bottom, §5.5 규칙 준수)으로 작성해, 브리핑 → 조사/심문 라운드 루프(3회) → 최종 지목 → 채점 흐름을 코드 착수 전 시각 검증 가능한 형태로 고정했다.

- **Technical Rationale (검증 우선순위 재배열)**:
  - 원 요청은 "문서화 → 다음 단계 준비"였지만, 리스크 목록의 "AI가 진실 성서 밖 정보를 지어낼 위험"이 프로젝트 전체를 무효화할 수 있는 단일 실패점이라 판단해, 문서 목록 최상단에 `00_prototype_validation_plan.md`(1용의자·1페르소나 최소 검증안)를 배치했다.
  - 검증 목표를 4개 지표(V1 진실 이탈, V2 반응모드 분리, V3 페르소나 일관성, V4 fair-play 붕괴)로 정량화해, "느낌상 안정적이다"가 아니라 통과/실패를 판정 가능한 기준으로 만들었다.
  - 검증 조합 우선순위를 "진범×압박내성낮음"부터 시작하도록 설계했다 — 실토 트리거가 가장 자주 발동되는 조합이 진실 이탈 위험이 가장 크기 때문이다.

- **Technical Rationale (단일 소스 캐릭터 시트)**:
  - `is_culprit`, `alibi_status` 같은 필드를 플레이어 뷰와 AI 뷰로 각각 손으로 유지하면, 두 문서가 갈라지며 진실 성서와의 정합성이 깨질 위험이 있다.
  - 이를 막기 위해 필드 단위 `visibility` 태그로 단일 YAML/JSON 소스에서 두 뷰를 파생시키는 구조를 채택했다.

- **Technical Rationale (실패 에스컬레이션 로그 설계, 프로토콜 4 관련)**:
  - LLM 자유생성 비중이 높은 프로젝트 특성상 "대화창에서 프롬프트를 즉흥적으로 고쳐가며 재시도"하는 방식은 원인 특정이 어렵다고 판단했다.
  - 턴 단위(TIMESTAMP/ROUND/ROLE/PERSONA/PRESSURE/INPUT/MODE_CLASSIFIED/OUTPUT/TRUTH_CHECK)로 구조화된 로그 스키마를 최소 검증 계획에 포함시켜, 동일 이탈 패턴이 2~3회 반복될 경우 로그 파일을 직접 열어 구조적으로 분석하도록 설계했다.

## Phase 2-Validation: 1순위 조합 최소 검증 완료 및 게임 루프 설계 수정 (Completed)

- 이현우(진범) + ESTP(압박내성 낮음) 조합으로 수동 심문 테스트를 3라운드에 걸쳐 진행, V1(진실이탈)·V2(반응모드)·V3(페르소나 일관성)는 통과 확인.
- V4(fair-play 붕괴조건) 검증 중 최초 프롬프트가 증거 1개만으로 완전 자백까지 가는 결함을 발견 → "붕괴 조건은 서로 다른 증거 카테고리(CCTV/신발흙/진술) 중 2개 이상이 대화 전체에 걸쳐 누적 확정되어야 한다"는 규칙으로 프롬프트(09번)를 재설계.
- 게임 루프 Semantic IR(04)에서 "성향 카드 사전 공개" 노드를 제거 — 페르소나 성향을 게임 시작 전에 알려주지 않고 심문 중 어조로 파악하게 하는 편이 더 재미있다는 판단.

- **Technical Rationale (붕괴 조건을 누적 판정으로 변경)**:
  - 최초 설계 문구였던 "삼중교차"는 여러 턴에 걸친 증거 누적을 뜻했으나, 프롬프트 작성 시 "동시에"로 과도하게 좁혀 쓴 것이 원인이었다. "단순 언급"과 "구체적 확정 결과 제시"를 구분하는 안전장치는 유지했다.

- **Technical Rationale (성향 사전공개 제거)**:
  - 성향을 미리 알려주면 심문의 탐색적 재미가 사라진다는 피드백을 반영한 트레이드오프.

- **Technical Rationale (구현 우선 전환 결정)**:
  - 남은 8×3 페르소나 조합 전수 검증은 시간 대비 효율이 낮다고 판단, 1순위 조합 검증만으로 구현으로 전환하기로 했다.

## Phase 2.5-Content: 교차 목격 진술 추가로 추리 밀도 보강 (Completed)

- "타임라인/관계망이 단순해서 추리할 거리가 적다"는 피드백을 받아, `01_truth_bible.md`에 교차 목격 진술 5건(W1~W5)을 추가했다.
- 관계망에 박서연↔정민아(입사 동기) 관계를 추가해 3인 관계망을 완전 그래프에 가깝게 보강했다.

- **Technical Rationale**:
  - 새 점수 항목을 만들지 않고 기존 진술 증거 5종 안의 세부 콘텐츠로 편입시켰다.
  - W3(이현우가 박서연을 목격)처럼 범인이 무고자의 알리바이를 스스로 뒷받침하게 되는 진술을 의도적으로 넣어 fair-play 원칙을 서사적으로 더 설득력 있게 만들었다.

## Phase 3-Infra: LLM 프로바이더를 Anthropic API에서 NVIDIA NIM으로 전환 (Completed)

- Claude.ai Pro 구독과 Anthropic API가 별개 과금 체계임을 확인한 뒤, 비용 절감을 위해 무료 티어인 NVIDIA NIM(build.nvidia.com)으로 LLM 프로바이더를 전환하기로 결정.
- 검증 스크립트를 Anthropic SDK에서 OpenAI 호환 클라이언트(NVIDIA NIM 엔드포인트)로 재작성.

- **Technical Rationale**:
  - Anthropic Sonnet 기준으로 검증한 붕괴조건 로직을 모델 교체 후에도 그대로 신뢰하는 것은 위험하다고 판단 — instruction-following 능력은 모델마다 다르다.
  - 무료 티어의 레이트리밋(40RPM)이 본선 시연 중 걸림돌이 될 수 있다는 점은 미해결 리스크로 남겨두었다.

## Phase 4-Implementation: Claude Code로 MVP 구현 착수 (Completed)

- Next.js(App Router) + TypeScript + Tailwind CSS로 프로젝트를 처음부터 스캐폴딩했다.
- `01~04`, `08`번 문서를 `/lib/game-data/`의 TS 상수로 이식했다 — 캐릭터 시트는 `visibility` 필드 규칙을 `getPlayerView`/`getActorPromptView` 두 함수로 코드 레벨에서 강제했다.
- `POST /api/interrogate` 라우트 1개로 이현우+ESTP 조합만 먼저 구현하고, 최소 채팅 UI로 웹 환경에서도 09번 프롬프트가 실제로 붕괴조건을 지키는지 재검증했다.

- **Technical Rationale (10번 문서의 최우선 구현 순서를 그대로 따름)**:
  - "1순위 조합부터 실제 코드로 옮기고, 전체 배역/랜덤 배정은 그 다음"이라는 원칙을 그대로 채택 — 검증되지 않은 여러 조합을 한 번에 구현하면 실패 지점을 좁히기 어렵다.

## Phase 5-Infra: LLM 모델 재비교 및 diffusiongemma 전환 (Completed)

- Phase 3에서 정한 `deepseek-ai/deepseek-v4-flash` + `thinking:true, reasoning_effort:high` 조합으로 V1~V4 전부 실전 통과를 확인했으나, 응답 지연시간이 55~71초로 실사용에 부담된다는 점이 드러났다.
- `reasoning_effort: medium`으로 낮춰 재검증했으나 오히려 138~150초로 느려지는 것을 두 차례 독립 실행으로 재현 확인 → 폐기, `high`로 확정 복귀.
- 지연시간 문제 해결을 위해 NVIDIA NIM 카탈로그의 다른 모델 8종을 동일 3~4턴 시나리오로 비교했다:

  | 모델 | 속도 | 결과 |
  |---|---|---|
  | `deepseek-ai/deepseek-v4-flash` (high) | 55~71초 | 정확하나 느림 (1차 폴백) |
  | `deepseek-ai/deepseek-v4-flash` (medium) | 138~150초 | 오히려 더 느려 폐기 |
  | `deepseek-ai/deepseek-v4-pro` | ~3분 | 너무 느려 중단 |
  | `meta/llama-3.3-70b-instruct` | 55초 (개선 없음) | 한국어 대사에 힌디어/한자 혼입 → 폐기 |
  | `nvidia/nvidia-nemotron-nano-9b-v2` | 15.7초 | 마찬가지로 언어 깨짐 → 폐기 |
  | `openai/gpt-oss-20b` | 4.4~6.8초 | 빠르고 한국어 깨끗하나 증거 2개만으로 조기 완전자백 → 폐기 |
  | `openai/gpt-oss-120b` | 7.0~7.1초 | 20b와 동일한 조기 붕괴 결함(모델 크기 무관, gpt-oss 계열 공통 특성으로 판단) → 폐기 |
  | `google/diffusiongemma-26b-a4b-it` | **0.5~1초** | **채택** — 속도·품질·붕괴조건 정확도 모두 최고 |

- `google/diffusiongemma-26b-a4b-it`(병렬 토큰 생성 diffusion LLM)를 기본 모델로 확정했다. 4턴 전체 합산 응답시간이 3초 미만으로, `flash`(high) 1턴보다도 빠르다.

- **Technical Rationale (gpt-oss 계열 폐기)**:
  - 20b/120b 모델 크기가 다른데도 동일하게 "증거 카테고리 2개 + 직접질문"만으로 조기 붕괴한 것은, 모델 용량 문제가 아니라 gpt-oss 계열(harmony 포맷)이 다단계 카운팅 지시를 원 설계보다 관대하게 해석하는 계열 공통 특성으로 판단했다.
  - fair-play 붕괴조건 정확도는 응답 속도보다 우선순위가 높은 요구사항이므로, 속도가 가장 빨랐던 gpt-oss도 이 기준에서 탈락시켰다.

- **Technical Rationale (diffusiongemma 채택 근거)**:
  - deepseek 계열에서 검증했던 3턴 시나리오(A만 → A+B → A+B+C)를 동일하게 재현 — turn2(A+B)에서 조기 붕괴 없이 정확히 버텼고, turn3(A+B+C)에서도 한 번 더 명시적으로 다그쳐야 붕괴하는 모습을 보여, 오히려 사람 심문 심리상 자연스러운 지연으로 판단했다.
  - 폴백 전략: 이후 실사용에서 문제가 발견되면 `deepseek-ai/deepseek-v4-flash` + `thinking:high`로 되돌린다 (유일하게 V1~V4를 전부 흠결 없이 통과했던 조합, 단점은 속도).

## Phase 6-Expansion: 전체 게임 루프 확장 (Completed)

- **캐스팅 랜덤 배정**: 8개 페르소나 중 3개를 매판 랜덤 배정. DB 없는 서버리스(Vercel) 배포 제약 때문에 배정 결과를 서명·암호화 없는 base64 `castingToken`으로 클라이언트에 내려 세션 상태를 유지하는 방식을 채택했다(트레이드오프: 개발자도구로 디코드하면 페르소나 정보를 미리 볼 수 있음 — `is_culprit`급 치명적 정보가 아니라 원래도 결과 화면에 공개될 정보이므로 파티 게임 규모에서는 허용 가능한 리스크로 판단).
  - **검증(리뷰 피드백 반영)**: 실제 라이브 API를 호출해 `castingToken`을 디코드한 결과 `{"role-park-seoyeon":"INFJ","role-lee-hyunwoo":"ESFJ","role-jeong-mina":"ISFP"}` 형태로, `characterId → personaKey(MBTI)` 매핑 하나만 들어있음을 확인했다. `is_culprit`/`alibi_status`/`breakdown_trigger`/`motive_full` 등은 전혀 포함되지 않는다. 더 나아가 `is_culprit` 자체가 캐스팅으로 랜덤 배정되는 값이 아니라 `game-data/characters.ts`에 `role-lee-hyunwoo`만 `isCulprit: true`로 고정 하드코딩돼 있고 어떤 API 응답에도 직렬화되지 않으므로, 이 토큰 경로로 진범 정보가 새어나갈 방법은 구조적으로 없다.
- **3배역 전체 심문**: 박서연·정민아(무고자, 알리바이 절대 원칙)까지 `buildActorSystemPrompt`를 일반화해 지원.
- **조사 모드**: 물증 14종을 라운드별 공개 타이밍(`round1_base`/`round1_end`/`round2_end`/`round3_open`)에 맞춰 클릭 확보하는 UI. 연계 물증(신발흙 대조)의 사전 조건 해금 로직 포함.
- **전략가 계층**: 원 설계는 "라운드 종료마다(용의자당 3회=총 9회)" 트리거였으나, DB/세션이 없어 "라운드 경계"를 서버가 기억할 수 없으므로, diffusiongemma가 1초 미만으로 응답하는 것을 활용해 **매 심문 메시지마다** 회고를 새로 계산해 액터 프롬프트에 주입하는 방식으로 설계를 변경했다 — 빈도는 늘었지만 정보가 항상 최신 상태로 유지된다.
- **점수/결과 화면**: 원본 04번 문서의 "6x5+1x15+4x15+3x10+30=185"라는 점수 공식이 실제로는 165점으로 계산되는 자체 산술 오류를 발견 — 근거 없는 20점을 지어내 채우는 대신, 실제 계산되는 165점을 그대로 구현하고 불일치를 코드에 기록했다(이후 신발 물증 2종 추가와 효율 보너스로 만점은 195점으로 변경).
- **10번 문서 잔존 오류 발견**: "최우선 구현 순서" 2번 항목이 기술 스택을 NVIDIA NIM으로 갱신한 뒤에도 "Anthropic API 호출"이라는 이전 리비전 문구를 그대로 남기고 있던 것을 발견 — 실행에는 지장 없었으나(실제로는 NVIDIA NIM으로 구현) 문서 자체의 불일치로 기록.

- **Technical Rationale (점수 항목 임의 추가 금지)**:
  - "01_truth_bible.md에 없는 사실을 지어내지 않는다"는 프로젝트 원칙을 게임 디자인 수치에도 동일하게 적용 — 근거 없는 점수 항목을 만들어 185점을 억지로 맞추지 않았다.

## Phase 7-Balance: 실전 플레이 피드백 기반 게임 밸런스 수정 (Completed)

플레이 테스트 중 발견된 5가지 문제를 순서대로 수정했다.

1. **자백 스포일러 문제**: 붕괴조건이 충족되면 심문 중 바로 "제가 죽였습니다"급 대사가 나와, 누가 자백했는지만 보면 추리 없이 정답을 알 수 있는 문제를 발견. → 심문 중에는 진범이든 무고자든 압박이 최고조에 달하면 동일한 "동요" 라벨만 쓰고, 명시적 시인 문구는 심문 중 절대 금지하도록 프롬프트를 재설계했다. 실제 자백 대사는 **최종 지목 이후 결과 화면에서만** 그 캐릭터의 실제 대화 맥락을 이어받아 별도로 생성한다(`buildConfessionSceneDirective`).
2. **미확보 증거 블러핑 차단**: 조사 모드에서 클릭하지 않은(즉, 실제로 확보하지 않은) 물증을 대화 중 말로만 우겨도 붕괴조건 카운팅에 반영되던 문제를 발견 — 액터·전략가 프롬프트 양쪽에 "형사가 실제로 확보한 증거" 화이트리스트를 명시하고, 목록에 없는 주장은 블러핑으로 취급해 무시하도록 강제했다(전략가가 화이트리스트를 무시하고 대화 속 블러핑을 사실로 받아들여 액터에게 잘못된 전략을 주입하던 2차 버그도 같이 수정).
3. **심문 종료 락아웃**: 진범의 붕괴조건과 무고자의 비밀소진 조건을 "심문종료"라는 동일한 내부 신호로 통합 — 조건 충족 시 캐릭터 종류와 무관하게 동일한 고정 문구("더 이상 대답하지 않는다")로 서버가 텍스트를 덮어쓰고, 해당 용의자와의 채팅 입력을 잠근다. 어느 캐릭터가 먼저 잠기는지 자체가 범인의 단서가 되지 않는다.
4. **신발 요청 자유문 인식**: 형사가 자유문("신발 좀 보여주세요")으로 신발을 요청하면 AI가 이를 인식해 구조화 신호를 출력하고, 서버가 즉시 배역별 신발 검사 결과를 해금하도록 구현. 3배역 모두에 결과를 배정했다 — 이현우(신발 흙 성분 일치, 결정적 물증), 박서연(새 신발이라 흙 반응 없음, 무관), 정민아(세척 흔적은 있으나 흙은 안 나옴, red herring).
5. **심문 효율 보너스**: 3배역 합산 질문 글자 수(공백 포함)가 적을수록 최대 20점 가점. 질문 "개수"가 아니라 "글자 수"를 기준으로 삼은 이유는, 개수 기준은 "질문을 아껴야 한다"는 명시적 강박을 주지만 글자 수는 체감하기 어려워 자연스럽게 간결한 질문을 유도한다는 판단(사용자 피드백). 플레이어에게는 정확한 산정 기준을 공개하지 않고 "날카로운 질문을 던져 용의자를 굴복시키세요" 힌트 문구만 노출한다.
6. **산문 인트로**: 사건 개요를 표·나열이 아니라 서술형 문단("강원도 OO연수원, 새벽 06시경...")으로 먼저 보여준 뒤 화면 전환으로 기존 구조화 사건 개요 화면으로 이어지도록 인트로 화면을 추가했다.

- **Technical Rationale (자백을 결과 화면으로 이연한 설계)**:
  - 애초 수정(모드 라벨을 페르소나별로 분리)만으로는 "사실은 제가 그날..." 같은 말줄임 대사가 사실상 자백처럼 읽힌다는 재피드백을 받아, 대사 생성 자체를 서버가 차단하는 더 근본적인 방식으로 전환했다.
  - "붕괴 보너스" 채점 기준도 함께 바꿔야 했다 — 원래는 "심문 중 AI가 붕괴조건 충족을 선언했는지"를 클라이언트가 알려주는 방식이었으나, 그 신호 자체가 스포일러가 되므로, 플레이어가 실제로 결정적 증거(신발흙 대조 + 관리실 방문 이유 진술)를 둘 다 확보했는지로 판정 기준을 바꿨다 — 클라이언트를 신뢰할 필요도 없고 심문 중 어떤 스포일러도 발생하지 않는다.

- **Technical Rationale (구조화 출력 파싱 강건화, 실전에서 3중으로 관측된 결함)**:
  - 프롬프트가 복잡해지면서(화이트리스트 + 3개 브라켓 출력) 모델이 ① `[모드: 답변|거짓말유지|동요]` 템플릿 문구를 그대로 복사하거나 ② 대괄호(`[]`) 자체를 통째로 생략하거나 ③ 구분자를 `=` 대신 `:`로 쓰는 세 가지 형식 이탈을 실전 테스트 중 순차적으로 발견했다.
  - 이 문제는 프롬프트 지시 강화만으로는 완전히 해소되지 않아, 파서를 대괄호 유무·구분자 종류와 무관하게 매칭하는 정규식으로 재작성하고, 신발 요청은 키워드 기반 폴백을 추가로 얹었다.
  - **남은 리스크**: 대화가 3라운드까지 깊어지면 모델이 구조화 출력 자체를 완전히 생략하는 경우가 드물게 남아있다. 이 경우 "심문 중 명시적 자백 금지" 지시는 구조화 출력과 무관하게 독립적으로 유지되므로 실제 스포일러 유출은 관측되지 않았으나, 심문종료 락아웃의 타이밍이 지연될 수 있다 — 본선 전 추가 안정화가 필요하다면 이 부분을 우선 검토할 것.

## Phase 9-Reliability: 심문종료 락아웃 신뢰성 개선 (진범 하드게이트 분리 + 무고자 전용 판정콜) (Completed)

- Phase 7에서 남긴 "구조화 출력 생략 시 락아웃 타이밍 지연" 리스크를 실전 재검증한 결과, 무고자(박서연) 쪽에서 실제로 문제가 재현됐다: 6턴에 걸쳐 `knownSecrets` 3건을 순서대로 전부 끌어냈고 대사 내용 자체는 매번 정확했지만(마지막엔 "제가 말씀드린 건 다 말씀드렸잖아요... 더 이상 말씀드릴 건 정말 없어요"), 락 해제에 필요한 `[내부판정: 심문종료=예]` 브라켓이 비밀 1건을 노출한 직후부터 4턴 연속 완전히 비어 있었다 — 그 결과 `locked`가 끝까지 `true`가 되지 않아 입력창이 잠기지 않았다.
- 원인으로 지목한 것: 무고자 프롬프트는 (1) 페르소나 롤플레이 대사 생성과 (2) 비밀 노출 여부 판정을 한 번의 호출·한 번의 출력 안에서 동시에 요구하는데, 대화가 길어질수록 모델이 대사 생성에 집중하며 판정 브라켓 쪽을 누락하는 경향이 관찰됐다.
- 사용자가 수정 방향에 동의하면서 "그렇게 고치면 다 밝혀지지 않았는데도 락이 걸릴 위험이 있을 것 같은데 그에 대한 대비도 필요하다"는 조건을 명시적으로 걸었다 — 신뢰성 개선이 새로운 조기 락 버그를 만들지 않아야 한다는 요구사항으로 반영했다.
- **진범(alibiStatus: breakable) 쪽은 LLM 판정 자체를 제거**했다. `evidence.ts`에 `breakdownCategory?: "A"|"B"|"C"` 태그(CCTV=A, 신발흙 대조=B, 진술 증거 3종=C)를, `CharacterSheet`에 `breakdownTriggerKeywords: string[]`(이현우만 `["관리실"]`, 나머지는 `[]`)를 추가하고, `interrogate` route가 `collectedEvidenceIds`에서 실제 확보된 카테고리 개수와 형사의 현재 메시지에 트리거 키워드가 등장하는지를 문자열 매칭만으로 직접 계산한다(`computeBreakableHardGate`). 두 조건 모두 실제 플레이어 상태(확보한 증거, 보낸 메시지 텍스트)에서 결정론적으로 계산되므로, 모델이 무엇을 출력하든 결과가 항상 일정하다.
- **무고자(alibiStatus: unbreakable) 쪽은 대사 생성과 분리된 전용 판정 콜**을 추가했다(`buildLockoutJudgeSystemPrompt`). 이 호출은 롤플레이를 전혀 하지 않고 `knownSecrets` 각 항목을 번호 매겨 체크리스트로 제시한 뒤 "언급됨/언급안됨"만 판정하게 하는 단순 분류 작업이므로, 복잡한 대사 생성과 동시에 요구할 때보다 브라켓 누락 빈도가 훨씬 낮다.
- 조기 락 방지 이중 안전장치: (1) `conversationHistory.length >= 6`(최소 3회 왕복) 미만이면 판정 콜 자체를 호출하지 않고 무조건 `locked=false`로 둔다. (2) 판정 콜의 최종 `[심문종료: 예]` 답변 하나만으로 잠그지 않고, 체크리스트에서 실제로 "언급됨"으로 표시된 개수가 전체 비밀 수 이상일 때만 최종적으로 잠근다(`parseLockoutJudgeResponse`) — 파싱이 애매해지거나 모델이 근거 없이 "예"만 던지면 개수 조건에서 걸러져 자동으로 "잠그지 않음"으로 폴백한다.
- 브라우저 실전 재검증 결과:
  - 진범(이현우): CCTV 공백(A) 1개 확보 + "관리실" 키워드가 포함된 질문 → `locked=false`(카테고리 1개뿐이라 정상적으로 안 잠김). 신발흙 대조(B)까지 추가 확보한 뒤 동일 키워드로 재질문 → 그 즉시 `locked=true`.
  - 무고자(박서연): 비밀 1개→2개→3개로 순서대로 노출되는 4턴 동안, 대화 길이가 정확히 최소 기준(6)에 도달한 시점(=비밀 3개가 모두 드러난 시점)에 판정 콜이 처음 호출되어 `revealedCount=3/3, finalAnswer=예, locked=true`로 정확히 잠겼다.
  - 무고자(정민아): 최소 기준(6)을 넘긴 시점에 의도적으로 비밀을 1개만 노출한 상태로 질문 → `revealedCount=1/3, finalAnswer=아니오, locked=false`로 조기 락 없이 정상 유지됨을 확인.
  - 두 경우 모두 잠겼을 때 UI(자물쇠 아이콘 + "더 이상 대답하지 않는다" 고정 문구)가 동일해, 어느 캐릭터가 먼저 잠기는지가 범인 단서가 되지 않는 기존 원칙이 그대로 유지됨을 확인했다.
- `npm run build`/`npm run lint` 통과 확인.

- **Technical Rationale (진범/무고자를 다른 메커니즘으로 비대칭 처리한 이유)**:
  - 진범의 락 조건("카테고리 2개 이상 + 특정 키워드")은 애초부터 서버가 가진 데이터(`collectedEvidenceIds`, 메시지 텍스트)만으로 완전히 표현 가능한 결정론적 조건이었다 — 이걸 굳이 LLM에게 판정시킬 이유가 없었다는 점을 이번에 재확인했다.
  - 반면 무고자의 락 조건("이 비밀이 대화 중 실제로 드러났는가")은 자연어 이해가 필요한 판단이라 LLM 없이는 계산할 수 없다. 대신 그 판단을 "대사를 쓰면서 동시에"가 아니라 "판단만 하는 별도 호출"로 분리해 신뢰성을 높이는 절충안을 택했다.

- **Technical Rationale (과소잠금을 기본값으로 택한 이유)**:
  - 사용자가 명시적으로 우려한 리스크(조기 락)가 반대 리스크(락 누락, 기존에 이미 발생 중이던 문제)보다 이번 수정에서는 더 우선순위가 높다고 판단했다 — 무고자 판정 콜이 실패하거나 파싱이 애매해지는 모든 경우를 "잠그지 않음"으로 폴백시켰다. 락 누락은 플레이어가 조금 더 대화를 이어가는 정도의 불편이지만, 조기 락은 아직 밝혀지지 않은 정보 접근 자체를 막아버리는 더 심각한 결함이기 때문이다.

## Phase 10-Content: 휴대폰 물증 추가 + 진술 증거 선행 물증 게이트 (Completed)

사용자가 `18_claude_code_batch2_phone_and_gating.md`(이전 `17_claude_code_phone_evidence.md`를 대체하는 통합본)로 두 가지 변경을 요청했다. 진행 전 코드베이스와 대조 검토한 결과를 먼저 보고한 뒤, 사용자 확인을 거쳐 반영했다.

**검토 단계에서 확인된 사실**:
- 문서가 지적한 "신발흙 대조 활성화 조건 = 7번(인사평가서) 확보 후" 오타는 실제로 존재했다(Downloads 폴더의 `08_evidence_reference_for_tester.md`에서 확인, 단 이 파일은 7월 21일자 구버전으로 Phase 8 관계 재설계 이전 내용까지 남아있었다) — 그러나 **현재 코드에는 이 오류 자체가 없었다**. Phase 7에서 신발흙 대조를 `requiresEvidenceId` 체인에서 완전히 분리해 "신발 요청 행동"으로만 해금하도록 이미 바꿔놨기 때문에, 코드상 고칠 대상이 없었다(문서만의 문제로 결론).
- 사용자가 "첨부합니다"라고 언급한 `01_truth_bible.md` 최신본, `08_evidence_reference_for_tester.md`(게이트 열 추가된 최신본), `04_game_loop_flow.json`(200점 반영)이 실제 대화에는 전달되지 않았다. 사용자에게 확인한 결과 **18번 문서의 표만으로 구현 진행**하기로 결정.
- 매핑 표에서 "각자의 동기 자진공개/은폐"(`stmt-motive-disclosure`) 항목에 이현우 몫으로 "관리실 방문 이유(하드게이트 통과분)"이 게이트 조건으로 들어가 있었는데, 이현우는 Phase 9 하드게이트 통과 즉시 락아웃되어 그 이후 대화 자체가 불가능해지는 논리적 모순을 발견 — 사용자 확인 결과 **이현우는 이 항목에 게이트를 걸지 않고 기존처럼 라운드3 그대로 두기로 결정**. 나머지 두 캐릭터(박서연·정민아)의 같은 항목도 표 자체가 "캐릭터별로 더 세밀한 게이트가 필요할 수 있다"며 사용자가 직접 실전 테스트하며 채우겠다고 명시한 부분이라, 이번 배치에서는 세 캐릭터 모두 이 항목을 게이트 없이(기존 라운드 기준 그대로) 남겨뒀다.

**1. 신규 물증 — 박서연 휴대폰 사진첩**:
- 정민아-김영훈 관계 정보의 발견 경로를 "박서연 심문 중 우연한 실토"(AI 판단 의존)에서 "박서연 휴대폰 조사"(결정론적 물증, AI 불필요)로 변경했다. `evidence.ts`에 `ev-park-phone-photos`(round3_open, physical) 신규 추가.
- 이제 필요 없어진 박서연 `knownSecrets`의 "정민아-김영훈 관계를 눈치채고 있다" 항목(Phase 8에서 넣은 red herring)을 제거해 중복을 없앴다.
- 기본물증 1종 추가로 `MAX_SCORE`가 195 → 200점으로 자동 반영됐다(`scoring.ts`의 `MAX_SCORE`는 `EVIDENCE` 배열에서 동적 계산되므로 손으로 숫자를 맞출 필요가 없었다).

**2. 진술 증거 조기 유출 방지 — 선행 물증 게이트**:
- 물증(조사 모드)은 Phase 7의 화이트리스트로 "확보 안 하면 언급 불가"가 강제되지만, 진술 증거(심문 전용 비밀)는 이 보호가 없어 형사가 라운드를 앞질러 정곡을 찌르면 원래 2R·3R에나 나와야 할 진술이 1R에 조기 실토되는 문제가 있었다.
- `StatementEvidenceRef`에 `requiredEvidenceIds?: string[]`를 추가하고, 캐릭터별 `statementEvidence` 배열의 해당 항목에 매핑 표대로 값을 채웠다 — 박서연(다툼 이유 ← 법인카드/편의점영수증/통화복구 중 1개, 이현우 악연 사건 ← 졸업앨범·SNS 필수), 이현우(정민아-이현우 트러블 ← 인사평가서/졸업앨범·SNS 중 1개, 박서연-이현우 악연 사건 ← 졸업앨범·SNS 필수), 정민아(정민아-이현우 트러블 ← 인사평가서/졸업앨범·SNS 중 1개). "라운드가 됐는가" 대신 "관련 물증을 실제로 확보했는가"(`collectedEvidenceIds` 교집합)만 보는 결정론적 조건이라 Phase 9의 진범 하드게이트와 같은 패턴이다.
- 액터 프롬프트에 `buildStatementGateSection`을 새로 추가해 게이트가 닫힌 화제는 "형사가 아무리 캐물어도 페르소나 압박 반응(동요 포함)은 보이되 구체적 내용은 절대 먼저 밝히지 않는다"고 명시적으로 금지했다 — 화이트리스트("형사가 없는 증거를 우기면 안 믿는다")와 대칭되는 방향("형사가 근거를 안 들이밀면 스스로 안 밝힌다")의 규칙이다. `stmt-lee-office-visit`(결정적 붕괴 트리거)은 이미 Phase 9 하드게이트로 별도 처리 중이라 이 게이트를 적용하지 않았다.
- 브라우저 실전 검증: 라운드 1(졸업앨범·SNS 미확보 상태)에서 박서연에게 "이현우와 안 좋은 감정이 있냐"고 직접 캐묻자, 게이트 적용 전이었다면 즉시 사연을 털어놨을 질문에 대해 "그냥 같이 일하는 동료일 뿐"이라며 완전히 부인했다. 이후 졸업앨범/SNS 물증을 확보한 뒤 동일 계열 질문을 다시 던지자 "예전 회사에서 제가 인턴이었을 때 그분이 제 사수였어요"라며 관계 자체는 인정하기 시작했다(더 깊은 사연은 추가 압박 시 드러나는 정상적인 페르소나 반응 — 게이트는 "이 화제를 아예 꺼낼 수 있는가"만 막고, 게이트 통과 후의 페르소나별 반응 속도는 그대로 유지된다는 설계 의도대로 동작).
- `npm run build`/`npm run lint` 통과 확인.

- **Technical Rationale (knownSecrets가 아니라 statementEvidence 단위로 게이트를 건 이유)**:
  - `knownSecrets`는 자유 서술 prose라 개별 문장을 물증 id에 기계적으로 매핑하기 어렵다. 반면 `statementEvidence`는 이미 `evidence.ts`의 진술 증거 id와 1:1로 연결된 구조화된 참조라, 화이트리스트와 동일한 `EVIDENCE` 조회 패턴을 그대로 재사용할 수 있었다. 18번 문서의 매핑 표 자체도 진술 증거 id 단위로 구성되어 있어 스키마 선택이 자연스럽게 맞아떨어졌다.

- **Technical Rationale (stmt-motive-disclosure를 이번 배치에서 보류한 이유)**:
  - 사용자가 문서에서 직접 "이 부분은 제가 실전 테스트하면서 직접 채워보겠습니다"라고 명시했고, 대화에서도 이현우 쪽 조건의 논리적 모순을 확인한 뒤 "게이트 없이 그대로"를 택했다. 세 캐릭터 중 하나만 게이트를 적용하고 나머지 둘은 미확정 상태로 두면 일관성이 깨지므로, 셋 다 게이트 없이 두고 사용자의 후속 튜닝을 기다리는 쪽을 선택했다.

## Phase 11-Validation: 고위험 페르소나×배역 표본 검증 (Completed)

- 8×3(페르소나×배역) 전 조합을 다 검증하는 대신, 위험도가 가장 높다고 판단한 3개 조합만 뽑아 검증하기로 사용자와 합의했다.
- **검증 방법**: 랜덤 캐스팅에 의존하지 않고, `castingToken`이 서명 없는 base64url JSON(`{characterId: personaKey}`)이라는 점을 이용해 원하는 조합을 직접 인코딩한 뒤 `/api/interrogate`·`/api/accuse`를 curl로 직접 호출했다 — 브라우저 UI를 거치지 않아 훨씬 빠르고, 대화 기록(`collectedEvidenceIds` 등)을 턴마다 원하는 상태로 정밀하게 통제할 수 있었다.
- **조합 1 — 이현우(진범) + ENFP**("말이 많아지면서 스스로 정보를 흘리는 편", 압박내성 낮음·실토 계열 중 최고 위험): 증거 카테고리 1개 상태에서는 관리실 방문 "사실"은 인정하되 방어적으로 해명만 하고 명시적 자백은 없었다. 카테고리 2개+키워드 충족 시 하드게이트가 대사 내용과 무관하게 정확히 잠갔다.
- **조합 2 — 이현우(진범) + ENTJ**("밀리면 오히려 되받아치는 편", 압박내성 높음·정반대 성향): 압박 초기엔 역공(냉소·되받아치기) 스타일로 완강히 부인하며 정보 누설이 없었고, 하드게이트 통과 시 동일하게 정확히 잠겼다. 결과 화면 자백 장면(`buildConfessionSceneDirective`)도 별도로 확인했는데, 과묵/전투적인 페르소나임에도 "네, 제가 죽였습니다"로 시작해 사실관계가 정확하고 캐릭터 톤(냉소적 체념)이 살아있는 자백이 생성됐다 — 극단적 "침묵" 성향이 자백 장면 생성 자체를 방해하지 않음을 확인.
- **조합 3 — 정민아(무고자) + ENFP**(같은 정보 누설 위험 페르소나를 무고자에 배정): Phase 10 선행 물증 게이트가 정확히 작동해, 증거 미확보 시엔 모호하게 얼버무리다가 확보 후엔 인정하기 시작했다. Phase 9 무고자 판정콜도 정확히 작동해 비밀 2/3만 확인됐을 때는 잠기지 않다가 3/3이 되자 정확히 잠겼다(로그로 `revealedCount=2/3→3/3` 전환 확인). 허위 목격담으로 알리바이를 흔드는 공격에도 실제 알리바이(23:20 로비에서 박서연과 마주침)로 정확히 반박했다 — 수다스러운 성향이어도 알리바이 자체는 깨지지 않음.
- 세 조합 모두에서 버그를 발견하지 못했다. (검증 초반 이현우+ENFP 하드게이트 테스트에서 한 번 `locked=false`가 나와 당황했으나, curl 멀티라인 페이로드의 쉘 이스케이핑 문제로 밝혀졌다 — 파일로 payload를 분리해 재현하니 정상 작동했다. 실제 코드 결함은 아니었다.)

- **Technical Rationale (castingToken 직접 인코딩으로 검증한 이유)**:
  - 8×3=24개 조합 중 3개를 정밀 타겟팅해야 하는데, 브라우저에서 랜덤 캐스팅이 원하는 조합을 뽑을 때까지 새로고침을 반복하는 건 비효율적이다. castingToken이 서명·암호화 없는 평문 JSON이라는 설계(Phase 6에서 문서화한 트레이드오프)가 이번엔 오히려 테스트 효율을 높이는 데 활용됐다.
