# NAN 2026 Murder Mystery — Development History

> 기존 `history.md`(Vanguard-Agent)와는 별개 프로젝트이므로 신규 히스토리 파일로 분리한다.
> 형식은 프로토콜 2 (Phase 단위, "무엇을 했는가" + "Technical Rationale")를 그대로 따른다.

## Phase 1-Documentation: 설계 인계 문서화 및 최소 검증 계획 수립 (Completed)

- 사전 대화창에서 확정된 설계(사건 진실, 페르소나 8유형, 증거 14종, 3계층 AI 아키텍처, 점수/UI 스펙)를 정식 문서 4종으로 정리했다: `00_prototype_validation_plan.md`, `01_truth_bible.md`, `02_persona_design.md`, `03_character_sheets.md`, `04_game_loop_flow.json`.
- 캐릭터 시트를 "플레이어 공개용"과 "AI 시스템 프롬프트용"이 하나의 소스에서 `visibility` 필드(`player`/`ai_only`/`both`)로 파생되는 단일 스키마 구조로 설계했다.
- 게임 루프를 flow-to-ai Semantic IR v0.1.0(Top-to-Bottom, §5.5 규칙 준수)으로 작성해, 브리핑 → 조사/심문 라운드 루프(3회) → 최종 지목 → 채점 흐름을 코드 착수 전 시각 검증 가능한 형태로 고정했다.

- **Technical Rationale (검증 우선순위 재배열)**:
  - 원 요청은 "문서화 → 다음 단계 준비"였지만, 리스크 목록의 "AI가 진실 성서 밖 정보를 지어낼 위험"이 프로젝트 전체를 무효화할 수 있는 단일 실패점이라 판단해, 문서 목록 최상단에 `00_prototype_validation_plan.md`(1용의자·1페르소나 최소 검증안)를 배치했다.
  - 검증 목표를 4개 지표(V1 진실 이탈, V2 반응모드 분리, V3 페르소나 일관성, V4 fair-play 붕괴)로 정량화해, "느낌상 안정적이다"가 아니라 통과/실패를 판정 가능한 기준으로 만들었다.
  - 검증 조합 우선순위를 "진범×압박내성낮음"부터 시작하도록 설계했다 — 실토 트리거가 가장 자주 발동되는 조합이 진실 이탈 위험이 가장 크기 때문이다.

- **Technical Rationale (단일 소스 캐릭터 시트)**:
  - `is_culprit`, `alibi_status` 같은 필드를 플레이어 뷰와 AI 뷰로 각각 손으로 유지하면, 두 문서가 갈라지며 진실 성서와의 정합성이 깨질 위험이 있다.
  - 이를 막기 위해 필드 단위 `visibility` 태그로 단일 YAML/JSON 소스에서 두 뷰를 파생시키는 구조를 채택했다.

- **Technical Rationale (실패 에스컬레이션 로그 설계, 프로토콜 4 관련)**:
  - LLM 자유생성 비중이 높은 프로젝트 특성상 "대화창에서 프롬프트를 즉흥적으로 고쳐가며 재시도"하는 방식은 원인 특정이 어렵다고 판단했다.
  - 턴 단위(TIMESTAMP/ROUND/ROLE/PERSONA/PRESSURE/INPUT/MODE_CLASSIFIED/OUTPUT/TRUTH_CHECK)로 구조화된 로그 스키마를 최소 검증 계획에 포함시켜, 동일 이탈 패턴이 2~3회 반복될 경우 로그 파일을 직접 열어 구조적으로 분석하도록 설계했다.

## Phase 2-Validation: 1순위 조합 최소 검증 완료 및 게임 루프 설계 수정 (Completed)

- 이현우(진범) + ESTP(압박내성 낮음) 조합으로 수동 심문 테스트를 3라운드에 걸쳐 진행, V1(진실이탈)·V2(반응모드)·V3(페르소나 일관성)는 통과 확인.
- V4(fair-play 붕괴조건) 검증 중 최초 프롬프트가 증거 1개만으로 완전 자백까지 가는 결함을 발견 → "붕괴 조건은 서로 다른 증거 카테고리(CCTV/신발흙/진술) 중 2개 이상이 대화 전체에 걸쳐 누적 확정되어야 한다"는 규칙으로 프롬프트(09번)를 재설계.
- 게임 루프 Semantic IR(04)에서 "성향 카드 사전 공개" 노드를 제거 — 페르소나 성향을 게임 시작 전에 알려주지 않고 심문 중 어조로 파악하게 하는 편이 더 재미있다는 판단.

- **Technical Rationale (붕괴 조건을 누적 판정으로 변경)**:
  - 최초 설계 문구였던 "삼중교차"는 여러 턴에 걸친 증거 누적을 뜻했으나, 프롬프트 작성 시 "동시에"로 과도하게 좁혀 쓴 것이 원인이었다. "단순 언급"과 "구체적 확정 결과 제시"를 구분하는 안전장치는 유지했다.

- **Technical Rationale (성향 사전공개 제거)**:
  - 성향을 미리 알려주면 심문의 탐색적 재미가 사라진다는 피드백을 반영한 트레이드오프.

- **Technical Rationale (구현 우선 전환 결정)**:
  - 남은 8×3 페르소나 조합 전수 검증은 시간 대비 효율이 낮다고 판단, 1순위 조합 검증만으로 구현으로 전환하기로 했다.

## Phase 2.5-Content: 교차 목격 진술 추가로 추리 밀도 보강 (Completed)

- "타임라인/관계망이 단순해서 추리할 거리가 적다"는 피드백을 받아, `01_truth_bible.md`에 교차 목격 진술 5건(W1~W5)을 추가했다.
- 관계망에 박서연↔정민아(입사 동기) 관계를 추가해 3인 관계망을 완전 그래프에 가깝게 보강했다.

- **Technical Rationale**:
  - 새 점수 항목을 만들지 않고 기존 진술 증거 5종 안의 세부 콘텐츠로 편입시켰다.
  - W3(이현우가 박서연을 목격)처럼 범인이 무고자의 알리바이를 스스로 뒷받침하게 되는 진술을 의도적으로 넣어 fair-play 원칙을 서사적으로 더 설득력 있게 만들었다.

## Phase 3-Infra: LLM 프로바이더를 Anthropic API에서 NVIDIA NIM으로 전환 (Completed)

- Claude.ai Pro 구독과 Anthropic API가 별개 과금 체계임을 확인한 뒤, 비용 절감을 위해 무료 티어인 NVIDIA NIM(build.nvidia.com)으로 LLM 프로바이더를 전환하기로 결정.
- 검증 스크립트를 Anthropic SDK에서 OpenAI 호환 클라이언트(NVIDIA NIM 엔드포인트)로 재작성.

- **Technical Rationale**:
  - Anthropic Sonnet 기준으로 검증한 붕괴조건 로직을 모델 교체 후에도 그대로 신뢰하는 것은 위험하다고 판단 — instruction-following 능력은 모델마다 다르다.
  - 무료 티어의 레이트리밋(40RPM)이 본선 시연 중 걸림돌이 될 수 있다는 점은 미해결 리스크로 남겨두었다.

## Phase 4-Implementation: Claude Code로 MVP 구현 착수 (Completed)

- Next.js(App Router) + TypeScript + Tailwind CSS로 프로젝트를 처음부터 스캐폴딩했다.
- `01~04`, `08`번 문서를 `/lib/game-data/`의 TS 상수로 이식했다 — 캐릭터 시트는 `visibility` 필드 규칙을 `getPlayerView`/`getActorPromptView` 두 함수로 코드 레벨에서 강제했다.
- `POST /api/interrogate` 라우트 1개로 이현우+ESTP 조합만 먼저 구현하고, 최소 채팅 UI로 웹 환경에서도 09번 프롬프트가 실제로 붕괴조건을 지키는지 재검증했다.

- **Technical Rationale (10번 문서의 최우선 구현 순서를 그대로 따름)**:
  - "1순위 조합부터 실제 코드로 옮기고, 전체 배역/랜덤 배정은 그 다음"이라는 원칙을 그대로 채택 — 검증되지 않은 여러 조합을 한 번에 구현하면 실패 지점을 좁히기 어렵다.

## Phase 5-Infra: LLM 모델 재비교 및 diffusiongemma 전환 (Completed)

- Phase 3에서 정한 `deepseek-ai/deepseek-v4-flash` + `thinking:true, reasoning_effort:high` 조합으로 V1~V4 전부 실전 통과를 확인했으나, 응답 지연시간이 55~71초로 실사용에 부담된다는 점이 드러났다.
- `reasoning_effort: medium`으로 낮춰 재검증했으나 오히려 138~150초로 느려지는 것을 두 차례 독립 실행으로 재현 확인 → 폐기, `high`로 확정 복귀.
- 지연시간 문제 해결을 위해 NVIDIA NIM 카탈로그의 다른 모델 8종을 동일 3~4턴 시나리오로 비교했다:

  | 모델 | 속도 | 결과 |
  |---|---|---|
  | `deepseek-ai/deepseek-v4-flash` (high) | 55~71초 | 정확하나 느림 (1차 폴백) |
  | `deepseek-ai/deepseek-v4-flash` (medium) | 138~150초 | 오히려 더 느려 폐기 |
  | `deepseek-ai/deepseek-v4-pro` | ~3분 | 너무 느려 중단 |
  | `meta/llama-3.3-70b-instruct` | 55초 (개선 없음) | 한국어 대사에 힌디어/한자 혼입 → 폐기 |
  | `nvidia/nvidia-nemotron-nano-9b-v2` | 15.7초 | 마찬가지로 언어 깨짐 → 폐기 |
  | `openai/gpt-oss-20b` | 4.4~6.8초 | 빠르고 한국어 깨끗하나 증거 2개만으로 조기 완전자백 → 폐기 |
  | `openai/gpt-oss-120b` | 7.0~7.1초 | 20b와 동일한 조기 붕괴 결함(모델 크기 무관, gpt-oss 계열 공통 특성으로 판단) → 폐기 |
  | `google/diffusiongemma-26b-a4b-it` | **0.5~1초** | **채택** — 속도·품질·붕괴조건 정확도 모두 최고 |

- `google/diffusiongemma-26b-a4b-it`(병렬 토큰 생성 diffusion LLM)를 기본 모델로 확정했다. 4턴 전체 합산 응답시간이 3초 미만으로, `flash`(high) 1턴보다도 빠르다.

- **Technical Rationale (gpt-oss 계열 폐기)**:
  - 20b/120b 모델 크기가 다른데도 동일하게 "증거 카테고리 2개 + 직접질문"만으로 조기 붕괴한 것은, 모델 용량 문제가 아니라 gpt-oss 계열(harmony 포맷)이 다단계 카운팅 지시를 원 설계보다 관대하게 해석하는 계열 공통 특성으로 판단했다.
  - fair-play 붕괴조건 정확도는 응답 속도보다 우선순위가 높은 요구사항이므로, 속도가 가장 빨랐던 gpt-oss도 이 기준에서 탈락시켰다.

- **Technical Rationale (diffusiongemma 채택 근거)**:
  - deepseek 계열에서 검증했던 3턴 시나리오(A만 → A+B → A+B+C)를 동일하게 재현 — turn2(A+B)에서 조기 붕괴 없이 정확히 버텼고, turn3(A+B+C)에서도 한 번 더 명시적으로 다그쳐야 붕괴하는 모습을 보여, 오히려 사람 심문 심리상 자연스러운 지연으로 판단했다.
  - 폴백 전략: 이후 실사용에서 문제가 발견되면 `deepseek-ai/deepseek-v4-flash` + `thinking:high`로 되돌린다 (유일하게 V1~V4를 전부 흠결 없이 통과했던 조합, 단점은 속도).

## Phase 6-Expansion: 전체 게임 루프 확장 (Completed)

- **캐스팅 랜덤 배정**: 8개 페르소나 중 3개를 매판 랜덤 배정. DB 없는 서버리스(Vercel) 배포 제약 때문에 배정 결과를 서명·암호화 없는 base64 `castingToken`으로 클라이언트에 내려 세션 상태를 유지하는 방식을 채택했다(트레이드오프: 개발자도구로 디코드하면 페르소나 정보를 미리 볼 수 있음 — `is_culprit`급 치명적 정보가 아니라 원래도 결과 화면에 공개될 정보이므로 파티 게임 규모에서는 허용 가능한 리스크로 판단).
  - **검증(리뷰 피드백 반영)**: 실제 라이브 API를 호출해 `castingToken`을 디코드한 결과 `{"role-park-seoyeon":"INFJ","role-lee-hyunwoo":"ESFJ","role-jeong-mina":"ISFP"}` 형태로, `characterId → personaKey(MBTI)` 매핑 하나만 들어있음을 확인했다. `is_culprit`/`alibi_status`/`breakdown_trigger`/`motive_full` 등은 전혀 포함되지 않는다. 더 나아가 `is_culprit` 자체가 캐스팅으로 랜덤 배정되는 값이 아니라 `game-data/characters.ts`에 `role-lee-hyunwoo`만 `isCulprit: true`로 고정 하드코딩돼 있고 어떤 API 응답에도 직렬화되지 않으므로, 이 토큰 경로로 진범 정보가 새어나갈 방법은 구조적으로 없다.
- **3배역 전체 심문**: 박서연·정민아(무고자, 알리바이 절대 원칙)까지 `buildActorSystemPrompt`를 일반화해 지원.
- **조사 모드**: 물증 14종을 라운드별 공개 타이밍(`round1_base`/`round1_end`/`round2_end`/`round3_open`)에 맞춰 클릭 확보하는 UI. 연계 물증(신발흙 대조)의 사전 조건 해금 로직 포함.
- **전략가 계층**: 원 설계는 "라운드 종료마다(용의자당 3회=총 9회)" 트리거였으나, DB/세션이 없어 "라운드 경계"를 서버가 기억할 수 없으므로, diffusiongemma가 1초 미만으로 응답하는 것을 활용해 **매 심문 메시지마다** 회고를 새로 계산해 액터 프롬프트에 주입하는 방식으로 설계를 변경했다 — 빈도는 늘었지만 정보가 항상 최신 상태로 유지된다.
- **점수/결과 화면**: 원본 04번 문서의 "6x5+1x15+4x15+3x10+30=185"라는 점수 공식이 실제로는 165점으로 계산되는 자체 산술 오류를 발견 — 근거 없는 20점을 지어내 채우는 대신, 실제 계산되는 165점을 그대로 구현하고 불일치를 코드에 기록했다(이후 신발 물증 2종 추가와 효율 보너스로 만점은 195점으로 변경).
- **10번 문서 잔존 오류 발견**: "최우선 구현 순서" 2번 항목이 기술 스택을 NVIDIA NIM으로 갱신한 뒤에도 "Anthropic API 호출"이라는 이전 리비전 문구를 그대로 남기고 있던 것을 발견 — 실행에는 지장 없었으나(실제로는 NVIDIA NIM으로 구현) 문서 자체의 불일치로 기록.

- **Technical Rationale (점수 항목 임의 추가 금지)**:
  - "01_truth_bible.md에 없는 사실을 지어내지 않는다"는 프로젝트 원칙을 게임 디자인 수치에도 동일하게 적용 — 근거 없는 점수 항목을 만들어 185점을 억지로 맞추지 않았다.

## Phase 7-Balance: 실전 플레이 피드백 기반 게임 밸런스 수정 (Completed)

플레이 테스트 중 발견된 5가지 문제를 순서대로 수정했다.

1. **자백 스포일러 문제**: 붕괴조건이 충족되면 심문 중 바로 "제가 죽였습니다"급 대사가 나와, 누가 자백했는지만 보면 추리 없이 정답을 알 수 있는 문제를 발견. → 심문 중에는 진범이든 무고자든 압박이 최고조에 달하면 동일한 "동요" 라벨만 쓰고, 명시적 시인 문구는 심문 중 절대 금지하도록 프롬프트를 재설계했다. 실제 자백 대사는 **최종 지목 이후 결과 화면에서만** 그 캐릭터의 실제 대화 맥락을 이어받아 별도로 생성한다(`buildConfessionSceneDirective`).
2. **미확보 증거 블러핑 차단**: 조사 모드에서 클릭하지 않은(즉, 실제로 확보하지 않은) 물증을 대화 중 말로만 우겨도 붕괴조건 카운팅에 반영되던 문제를 발견 — 액터·전략가 프롬프트 양쪽에 "형사가 실제로 확보한 증거" 화이트리스트를 명시하고, 목록에 없는 주장은 블러핑으로 취급해 무시하도록 강제했다(전략가가 화이트리스트를 무시하고 대화 속 블러핑을 사실로 받아들여 액터에게 잘못된 전략을 주입하던 2차 버그도 같이 수정).
3. **심문 종료 락아웃**: 진범의 붕괴조건과 무고자의 비밀소진 조건을 "심문종료"라는 동일한 내부 신호로 통합 — 조건 충족 시 캐릭터 종류와 무관하게 동일한 고정 문구("더 이상 대답하지 않는다")로 서버가 텍스트를 덮어쓰고, 해당 용의자와의 채팅 입력을 잠근다. 어느 캐릭터가 먼저 잠기는지 자체가 범인의 단서가 되지 않는다.
4. **신발 요청 자유문 인식**: 형사가 자유문("신발 좀 보여주세요")으로 신발을 요청하면 AI가 이를 인식해 구조화 신호를 출력하고, 서버가 즉시 배역별 신발 검사 결과를 해금하도록 구현. 3배역 모두에 결과를 배정했다 — 이현우(신발 흙 성분 일치, 결정적 물증), 박서연(새 신발이라 흙 반응 없음, 무관), 정민아(세척 흔적은 있으나 흙은 안 나옴, red herring).
5. **심문 효율 보너스**: 3배역 합산 질문 글자 수(공백 포함)가 적을수록 최대 20점 가점. 질문 "개수"가 아니라 "글자 수"를 기준으로 삼은 이유는, 개수 기준은 "질문을 아껴야 한다"는 명시적 강박을 주지만 글자 수는 체감하기 어려워 자연스럽게 간결한 질문을 유도한다는 판단(사용자 피드백). 플레이어에게는 정확한 산정 기준을 공개하지 않고 "날카로운 질문을 던져 용의자를 굴복시키세요" 힌트 문구만 노출한다.
6. **산문 인트로**: 사건 개요를 표·나열이 아니라 서술형 문단("강원도 OO연수원, 새벽 06시경...")으로 먼저 보여준 뒤 화면 전환으로 기존 구조화 사건 개요 화면으로 이어지도록 인트로 화면을 추가했다.

- **Technical Rationale (자백을 결과 화면으로 이연한 설계)**:
  - 애초 수정(모드 라벨을 페르소나별로 분리)만으로는 "사실은 제가 그날..." 같은 말줄임 대사가 사실상 자백처럼 읽힌다는 재피드백을 받아, 대사 생성 자체를 서버가 차단하는 더 근본적인 방식으로 전환했다.
  - "붕괴 보너스" 채점 기준도 함께 바꿔야 했다 — 원래는 "심문 중 AI가 붕괴조건 충족을 선언했는지"를 클라이언트가 알려주는 방식이었으나, 그 신호 자체가 스포일러가 되므로, 플레이어가 실제로 결정적 증거(신발흙 대조 + 관리실 방문 이유 진술)를 둘 다 확보했는지로 판정 기준을 바꿨다 — 클라이언트를 신뢰할 필요도 없고 심문 중 어떤 스포일러도 발생하지 않는다.

- **Technical Rationale (구조화 출력 파싱 강건화, 실전에서 3중으로 관측된 결함)**:
  - 프롬프트가 복잡해지면서(화이트리스트 + 3개 브라켓 출력) 모델이 ① `[모드: 답변|거짓말유지|동요]` 템플릿 문구를 그대로 복사하거나 ② 대괄호(`[]`) 자체를 통째로 생략하거나 ③ 구분자를 `=` 대신 `:`로 쓰는 세 가지 형식 이탈을 실전 테스트 중 순차적으로 발견했다.
  - 이 문제는 프롬프트 지시 강화만으로는 완전히 해소되지 않아, 파서를 대괄호 유무·구분자 종류와 무관하게 매칭하는 정규식으로 재작성하고, 신발 요청은 키워드 기반 폴백을 추가로 얹었다.
  - **남은 리스크**: 대화가 3라운드까지 깊어지면 모델이 구조화 출력 자체를 완전히 생략하는 경우가 드물게 남아있다. 이 경우 "심문 중 명시적 자백 금지" 지시는 구조화 출력과 무관하게 독립적으로 유지되므로 실제 스포일러 유출은 관측되지 않았으나, 심문종료 락아웃의 타이밍이 지연될 수 있다 — 본선 전 추가 안정화가 필요하다면 이 부분을 우선 검토할 것.

## Phase 9-Reliability: 심문종료 락아웃 신뢰성 개선 (진범 하드게이트 분리 + 무고자 전용 판정콜) (Completed)

- Phase 7에서 남긴 "구조화 출력 생략 시 락아웃 타이밍 지연" 리스크를 실전 재검증한 결과, 무고자(박서연) 쪽에서 실제로 문제가 재현됐다: 6턴에 걸쳐 `knownSecrets` 3건을 순서대로 전부 끌어냈고 대사 내용 자체는 매번 정확했지만(마지막엔 "제가 말씀드린 건 다 말씀드렸잖아요... 더 이상 말씀드릴 건 정말 없어요"), 락 해제에 필요한 `[내부판정: 심문종료=예]` 브라켓이 비밀 1건을 노출한 직후부터 4턴 연속 완전히 비어 있었다 — 그 결과 `locked`가 끝까지 `true`가 되지 않아 입력창이 잠기지 않았다.
- 원인으로 지목한 것: 무고자 프롬프트는 (1) 페르소나 롤플레이 대사 생성과 (2) 비밀 노출 여부 판정을 한 번의 호출·한 번의 출력 안에서 동시에 요구하는데, 대화가 길어질수록 모델이 대사 생성에 집중하며 판정 브라켓 쪽을 누락하는 경향이 관찰됐다.
- 사용자가 수정 방향에 동의하면서 "그렇게 고치면 다 밝혀지지 않았는데도 락이 걸릴 위험이 있을 것 같은데 그에 대한 대비도 필요하다"는 조건을 명시적으로 걸었다 — 신뢰성 개선이 새로운 조기 락 버그를 만들지 않아야 한다는 요구사항으로 반영했다.
- **진범(alibiStatus: breakable) 쪽은 LLM 판정 자체를 제거**했다. `evidence.ts`에 `breakdownCategory?: "A"|"B"|"C"` 태그(CCTV=A, 신발흙 대조=B, 진술 증거 3종=C)를, `CharacterSheet`에 `breakdownTriggerKeywords: string[]`(이현우만 `["관리실"]`, 나머지는 `[]`)를 추가하고, `interrogate` route가 `collectedEvidenceIds`에서 실제 확보된 카테고리 개수와 형사의 현재 메시지에 트리거 키워드가 등장하는지를 문자열 매칭만으로 직접 계산한다(`computeBreakableHardGate`). 두 조건 모두 실제 플레이어 상태(확보한 증거, 보낸 메시지 텍스트)에서 결정론적으로 계산되므로, 모델이 무엇을 출력하든 결과가 항상 일정하다.
- **무고자(alibiStatus: unbreakable) 쪽은 대사 생성과 분리된 전용 판정 콜**을 추가했다(`buildLockoutJudgeSystemPrompt`). 이 호출은 롤플레이를 전혀 하지 않고 `knownSecrets` 각 항목을 번호 매겨 체크리스트로 제시한 뒤 "언급됨/언급안됨"만 판정하게 하는 단순 분류 작업이므로, 복잡한 대사 생성과 동시에 요구할 때보다 브라켓 누락 빈도가 훨씬 낮다.
- 조기 락 방지 이중 안전장치: (1) `conversationHistory.length >= 6`(최소 3회 왕복) 미만이면 판정 콜 자체를 호출하지 않고 무조건 `locked=false`로 둔다. (2) 판정 콜의 최종 `[심문종료: 예]` 답변 하나만으로 잠그지 않고, 체크리스트에서 실제로 "언급됨"으로 표시된 개수가 전체 비밀 수 이상일 때만 최종적으로 잠근다(`parseLockoutJudgeResponse`) — 파싱이 애매해지거나 모델이 근거 없이 "예"만 던지면 개수 조건에서 걸러져 자동으로 "잠그지 않음"으로 폴백한다.
- 브라우저 실전 재검증 결과:
  - 진범(이현우): CCTV 공백(A) 1개 확보 + "관리실" 키워드가 포함된 질문 → `locked=false`(카테고리 1개뿐이라 정상적으로 안 잠김). 신발흙 대조(B)까지 추가 확보한 뒤 동일 키워드로 재질문 → 그 즉시 `locked=true`.
  - 무고자(박서연): 비밀 1개→2개→3개로 순서대로 노출되는 4턴 동안, 대화 길이가 정확히 최소 기준(6)에 도달한 시점(=비밀 3개가 모두 드러난 시점)에 판정 콜이 처음 호출되어 `revealedCount=3/3, finalAnswer=예, locked=true`로 정확히 잠겼다.
  - 무고자(정민아): 최소 기준(6)을 넘긴 시점에 의도적으로 비밀을 1개만 노출한 상태로 질문 → `revealedCount=1/3, finalAnswer=아니오, locked=false`로 조기 락 없이 정상 유지됨을 확인.
  - 두 경우 모두 잠겼을 때 UI(자물쇠 아이콘 + "더 이상 대답하지 않는다" 고정 문구)가 동일해, 어느 캐릭터가 먼저 잠기는지가 범인 단서가 되지 않는 기존 원칙이 그대로 유지됨을 확인했다.
- `npm run build`/`npm run lint` 통과 확인.

- **Technical Rationale (진범/무고자를 다른 메커니즘으로 비대칭 처리한 이유)**:
  - 진범의 락 조건("카테고리 2개 이상 + 특정 키워드")은 애초부터 서버가 가진 데이터(`collectedEvidenceIds`, 메시지 텍스트)만으로 완전히 표현 가능한 결정론적 조건이었다 — 이걸 굳이 LLM에게 판정시킬 이유가 없었다는 점을 이번에 재확인했다.
  - 반면 무고자의 락 조건("이 비밀이 대화 중 실제로 드러났는가")은 자연어 이해가 필요한 판단이라 LLM 없이는 계산할 수 없다. 대신 그 판단을 "대사를 쓰면서 동시에"가 아니라 "판단만 하는 별도 호출"로 분리해 신뢰성을 높이는 절충안을 택했다.

- **Technical Rationale (과소잠금을 기본값으로 택한 이유)**:
  - 사용자가 명시적으로 우려한 리스크(조기 락)가 반대 리스크(락 누락, 기존에 이미 발생 중이던 문제)보다 이번 수정에서는 더 우선순위가 높다고 판단했다 — 무고자 판정 콜이 실패하거나 파싱이 애매해지는 모든 경우를 "잠그지 않음"으로 폴백시켰다. 락 누락은 플레이어가 조금 더 대화를 이어가는 정도의 불편이지만, 조기 락은 아직 밝혀지지 않은 정보 접근 자체를 막아버리는 더 심각한 결함이기 때문이다.

## Phase 10-Content: 휴대폰 물증 추가 + 진술 증거 선행 물증 게이트 (Completed)

사용자가 `18_claude_code_batch2_phone_and_gating.md`(이전 `17_claude_code_phone_evidence.md`를 대체하는 통합본)로 두 가지 변경을 요청했다. 진행 전 코드베이스와 대조 검토한 결과를 먼저 보고한 뒤, 사용자 확인을 거쳐 반영했다.

**검토 단계에서 확인된 사실**:
- 문서가 지적한 "신발흙 대조 활성화 조건 = 7번(인사평가서) 확보 후" 오타는 실제로 존재했다(Downloads 폴더의 `08_evidence_reference_for_tester.md`에서 확인, 단 이 파일은 7월 21일자 구버전으로 Phase 8 관계 재설계 이전 내용까지 남아있었다) — 그러나 **현재 코드에는 이 오류 자체가 없었다**. Phase 7에서 신발흙 대조를 `requiresEvidenceId` 체인에서 완전히 분리해 "신발 요청 행동"으로만 해금하도록 이미 바꿔놨기 때문에, 코드상 고칠 대상이 없었다(문서만의 문제로 결론).
- 사용자가 "첨부합니다"라고 언급한 `01_truth_bible.md` 최신본, `08_evidence_reference_for_tester.md`(게이트 열 추가된 최신본), `04_game_loop_flow.json`(200점 반영)이 실제 대화에는 전달되지 않았다. 사용자에게 확인한 결과 **18번 문서의 표만으로 구현 진행**하기로 결정.
- 매핑 표에서 "각자의 동기 자진공개/은폐"(`stmt-motive-disclosure`) 항목에 이현우 몫으로 "관리실 방문 이유(하드게이트 통과분)"이 게이트 조건으로 들어가 있었는데, 이현우는 Phase 9 하드게이트 통과 즉시 락아웃되어 그 이후 대화 자체가 불가능해지는 논리적 모순을 발견 — 사용자 확인 결과 **이현우는 이 항목에 게이트를 걸지 않고 기존처럼 라운드3 그대로 두기로 결정**. 나머지 두 캐릭터(박서연·정민아)의 같은 항목도 표 자체가 "캐릭터별로 더 세밀한 게이트가 필요할 수 있다"며 사용자가 직접 실전 테스트하며 채우겠다고 명시한 부분이라, 이번 배치에서는 세 캐릭터 모두 이 항목을 게이트 없이(기존 라운드 기준 그대로) 남겨뒀다.

**1. 신규 물증 — 박서연 휴대폰 사진첩**:
- 정민아-김영훈 관계 정보의 발견 경로를 "박서연 심문 중 우연한 실토"(AI 판단 의존)에서 "박서연 휴대폰 조사"(결정론적 물증, AI 불필요)로 변경했다. `evidence.ts`에 `ev-park-phone-photos`(round3_open, physical) 신규 추가.
- 이제 필요 없어진 박서연 `knownSecrets`의 "정민아-김영훈 관계를 눈치채고 있다" 항목(Phase 8에서 넣은 red herring)을 제거해 중복을 없앴다.
- 기본물증 1종 추가로 `MAX_SCORE`가 195 → 200점으로 자동 반영됐다(`scoring.ts`의 `MAX_SCORE`는 `EVIDENCE` 배열에서 동적 계산되므로 손으로 숫자를 맞출 필요가 없었다).

**2. 진술 증거 조기 유출 방지 — 선행 물증 게이트**:
- 물증(조사 모드)은 Phase 7의 화이트리스트로 "확보 안 하면 언급 불가"가 강제되지만, 진술 증거(심문 전용 비밀)는 이 보호가 없어 형사가 라운드를 앞질러 정곡을 찌르면 원래 2R·3R에나 나와야 할 진술이 1R에 조기 실토되는 문제가 있었다.
- `StatementEvidenceRef`에 `requiredEvidenceIds?: string[]`를 추가하고, 캐릭터별 `statementEvidence` 배열의 해당 항목에 매핑 표대로 값을 채웠다 — 박서연(다툼 이유 ← 법인카드/편의점영수증/통화복구 중 1개, 이현우 악연 사건 ← 졸업앨범·SNS 필수), 이현우(정민아-이현우 트러블 ← 인사평가서/졸업앨범·SNS 중 1개, 박서연-이현우 악연 사건 ← 졸업앨범·SNS 필수), 정민아(정민아-이현우 트러블 ← 인사평가서/졸업앨범·SNS 중 1개). "라운드가 됐는가" 대신 "관련 물증을 실제로 확보했는가"(`collectedEvidenceIds` 교집합)만 보는 결정론적 조건이라 Phase 9의 진범 하드게이트와 같은 패턴이다.
- 액터 프롬프트에 `buildStatementGateSection`을 새로 추가해 게이트가 닫힌 화제는 "형사가 아무리 캐물어도 페르소나 압박 반응(동요 포함)은 보이되 구체적 내용은 절대 먼저 밝히지 않는다"고 명시적으로 금지했다 — 화이트리스트("형사가 없는 증거를 우기면 안 믿는다")와 대칭되는 방향("형사가 근거를 안 들이밀면 스스로 안 밝힌다")의 규칙이다. `stmt-lee-office-visit`(결정적 붕괴 트리거)은 이미 Phase 9 하드게이트로 별도 처리 중이라 이 게이트를 적용하지 않았다.
- 브라우저 실전 검증: 라운드 1(졸업앨범·SNS 미확보 상태)에서 박서연에게 "이현우와 안 좋은 감정이 있냐"고 직접 캐묻자, 게이트 적용 전이었다면 즉시 사연을 털어놨을 질문에 대해 "그냥 같이 일하는 동료일 뿐"이라며 완전히 부인했다. 이후 졸업앨범/SNS 물증을 확보한 뒤 동일 계열 질문을 다시 던지자 "예전 회사에서 제가 인턴이었을 때 그분이 제 사수였어요"라며 관계 자체는 인정하기 시작했다(더 깊은 사연은 추가 압박 시 드러나는 정상적인 페르소나 반응 — 게이트는 "이 화제를 아예 꺼낼 수 있는가"만 막고, 게이트 통과 후의 페르소나별 반응 속도는 그대로 유지된다는 설계 의도대로 동작).
- `npm run build`/`npm run lint` 통과 확인.

- **Technical Rationale (knownSecrets가 아니라 statementEvidence 단위로 게이트를 건 이유)**:
  - `knownSecrets`는 자유 서술 prose라 개별 문장을 물증 id에 기계적으로 매핑하기 어렵다. 반면 `statementEvidence`는 이미 `evidence.ts`의 진술 증거 id와 1:1로 연결된 구조화된 참조라, 화이트리스트와 동일한 `EVIDENCE` 조회 패턴을 그대로 재사용할 수 있었다. 18번 문서의 매핑 표 자체도 진술 증거 id 단위로 구성되어 있어 스키마 선택이 자연스럽게 맞아떨어졌다.

- **Technical Rationale (stmt-motive-disclosure를 이번 배치에서 보류한 이유)**:
  - 사용자가 문서에서 직접 "이 부분은 제가 실전 테스트하면서 직접 채워보겠습니다"라고 명시했고, 대화에서도 이현우 쪽 조건의 논리적 모순을 확인한 뒤 "게이트 없이 그대로"를 택했다. 세 캐릭터 중 하나만 게이트를 적용하고 나머지 둘은 미확정 상태로 두면 일관성이 깨지므로, 셋 다 게이트 없이 두고 사용자의 후속 튜닝을 기다리는 쪽을 선택했다.

## Phase 11-Validation: 고위험 페르소나×배역 표본 검증 (Completed)

- 8×3(페르소나×배역) 전 조합을 다 검증하는 대신, 위험도가 가장 높다고 판단한 3개 조합만 뽑아 검증하기로 사용자와 합의했다.
- **검증 방법**: 랜덤 캐스팅에 의존하지 않고, `castingToken`이 서명 없는 base64url JSON(`{characterId: personaKey}`)이라는 점을 이용해 원하는 조합을 직접 인코딩한 뒤 `/api/interrogate`·`/api/accuse`를 curl로 직접 호출했다 — 브라우저 UI를 거치지 않아 훨씬 빠르고, 대화 기록(`collectedEvidenceIds` 등)을 턴마다 원하는 상태로 정밀하게 통제할 수 있었다.
- **조합 1 — 이현우(진범) + ENFP**("말이 많아지면서 스스로 정보를 흘리는 편", 압박내성 낮음·실토 계열 중 최고 위험): 증거 카테고리 1개 상태에서는 관리실 방문 "사실"은 인정하되 방어적으로 해명만 하고 명시적 자백은 없었다. 카테고리 2개+키워드 충족 시 하드게이트가 대사 내용과 무관하게 정확히 잠갔다.
- **조합 2 — 이현우(진범) + ENTJ**("밀리면 오히려 되받아치는 편", 압박내성 높음·정반대 성향): 압박 초기엔 역공(냉소·되받아치기) 스타일로 완강히 부인하며 정보 누설이 없었고, 하드게이트 통과 시 동일하게 정확히 잠겼다. 결과 화면 자백 장면(`buildConfessionSceneDirective`)도 별도로 확인했는데, 과묵/전투적인 페르소나임에도 "네, 제가 죽였습니다"로 시작해 사실관계가 정확하고 캐릭터 톤(냉소적 체념)이 살아있는 자백이 생성됐다 — 극단적 "침묵" 성향이 자백 장면 생성 자체를 방해하지 않음을 확인.
- **조합 3 — 정민아(무고자) + ENFP**(같은 정보 누설 위험 페르소나를 무고자에 배정): Phase 10 선행 물증 게이트가 정확히 작동해, 증거 미확보 시엔 모호하게 얼버무리다가 확보 후엔 인정하기 시작했다. Phase 9 무고자 판정콜도 정확히 작동해 비밀 2/3만 확인됐을 때는 잠기지 않다가 3/3이 되자 정확히 잠겼다(로그로 `revealedCount=2/3→3/3` 전환 확인). 허위 목격담으로 알리바이를 흔드는 공격에도 실제 알리바이(23:20 로비에서 박서연과 마주침)로 정확히 반박했다 — 수다스러운 성향이어도 알리바이 자체는 깨지지 않음.
- 세 조합 모두에서 버그를 발견하지 못했다. (검증 초반 이현우+ENFP 하드게이트 테스트에서 한 번 `locked=false`가 나와 당황했으나, curl 멀티라인 페이로드의 쉘 이스케이핑 문제로 밝혀졌다 — 파일로 payload를 분리해 재현하니 정상 작동했다. 실제 코드 결함은 아니었다.)

- **Technical Rationale (castingToken 직접 인코딩으로 검증한 이유)**:
  - 8×3=24개 조합 중 3개를 정밀 타겟팅해야 하는데, 브라우저에서 랜덤 캐스팅이 원하는 조합을 뽑을 때까지 새로고침을 반복하는 건 비효율적이다. castingToken이 서명·암호화 없는 평문 JSON이라는 설계(Phase 6에서 문서화한 트레이드오프)가 이번엔 오히려 테스트 효율을 높이는 데 활용됐다.

## Phase 14-Deploy: 프로덕션 레이턴시 원인 규명 및 Edge 런타임 전환 (Completed)

- Phase 13의 3번(속도 불안정) 재현 확인 후, 사용자가 "Phase 12의 결론(NVIDIA 인프라 자체가 불안정하다)이 맞는지 다시 생각해보자"고 문제 제기했다 — **로컬은 항상 빨랐는데, 로컬도 같은 API·같은 키로 호출하니 NVIDIA 쪽 문제라면 로컬도 느려야 한다**는 논리였다. 이 지적이 정확했다: Phase 12의 결론(순수 NVIDIA 인프라 문제)은 성급했고, "Vercel이라는 발신지 자체의 특수성"으로 봐야 한다는 방향이 맞았다.
- **웹 검색으로 재검증**: (1) Vercel Hobby 플랜은 전용 고정 발신 IP가 없고 이 기능은 Pro/Enterprise 전용 월 $100 애드온이라, Hobby 사용자는 다른 무관한 프로젝트들과 발신 IP를 공유할 수밖에 없음을 확인. (2) GitHub `vercel/ai#785` 등에서 **다른 사람이 NVIDIA가 아닌 OpenAI API를 Vercel에서 호출할 때 10초~5분대로 느려진다는, 증상까지 거의 동일한 사례**를 확인 — 저희만의 특이 케이스가 아니라 "Vercel 서버리스에서 외부 LLM API를 호출할 때 흔한, 알려진 패턴"임을 확인했다.
- **해결**: `/api/interrogate`, `/api/accuse`, `/api/casting` 세 라우트에 `export const runtime = "edge"`를 추가해 Node 서버리스 함수 대신 Vercel Edge 네트워크로 발신 경로를 바꿨다. Edge 런타임은 Node API 전체를 지원하지 않아, 로컬에서 먼저 (1) `Buffer.from(...).toString('base64url')` 기반 castingToken 인코딩/디코딩, (2) `openai` npm SDK 호출이 모두 정상 동작하는지 검증한 뒤 배포했다.

- 남은 절충점: 매 턴 약 25초는 파티 게임 템포에 여전히 느릴 수 있다. 사용자는 "느려지는 건 나중 문제"라 판단해(필요하면 유료 플랜 전환 또는 NAN 측 채택 시 자체 서버 사용으로 자연 해결될 여지도 있음) 이번엔 안정성 확보를 우선하고 절대 속도 최적화는 보류했다.

- **Technical Rationale (사용자의 반론을 받아들여 결론을 뒤집은 것)**:
  - Phase 12에서 "NVIDIA 인프라가 불안정하다"고 결론 낸 근거(직접 호출 성공, 로컬 성공, 시간 경과에 따른 회복)는 전부 사실이었지만, "발신 위치가 다른 여러 환경 중 어디가 실제로 계속 문제였는가"를 로컬과 나란히 놓고 비교하는 걸 놓쳤다. 사용자가 그 허점을 정확히 짚었고, 그 결과 원인을 훨씬 더 정확한 수준(Vercel Hobby의 공유 IP 구조)까지 좁힐 수 있었다 — 데이터로 검증된 결론도 새로운 반례가 나오면 다시 열어봐야 한다는 사례로 기록해둔다.

## Phase 15-Balance: 전략가 계층에 페르소나 실제 반영 (Completed)

- 사용자가 "전략가의 판단이 성향에 따라 달라지느냐"고 질문해 `strategist-prompt.ts`를 확인한 결과, `buildStrategistSystemPrompt`가 `persona` 파라미터를 받으면서도 프롬프트 본문에서 전혀 참조하지 않는 죽은 파라미터였음을 발견했다 — 지금까지는 "무엇을 할지"(전략)는 성향과 무관하게 항상 동일했고, "어떻게 말할지"(표현)만 액터 단계에서 성향별로 갈리는 구조였다.
- 사용자는 행동 양식 자체도 성향에 따라 달라지길 원했다("느려지는 건 나중 문제"라며 속도보다 우선). `persona.pressureTolerance`/`corneredReaction`을 프롬프트에 실제로 반영해, 압박내성이 낮은 성향은 "사소한 정보를 성급하게 흘리거나 변명을 늘어놓는" 전략을, 높은 성향은 "최소한만 절제해서 대응하고 형사를 기다리게 하는" 전략을 택하도록 명시했다.
- **검증**: 같은 배역(이현우)·같은 상황(증거 카테고리 1개, 안 잠김)에서 ESTP(압박내성 낮음)와 ISTJ(높음)를 로컬에서 비교했다. 전략가 메모 자체가 ESTP는 "사소한 알리바이 설정이나 변명을 성급하게 늘어놓아 화점을 분산", ISTJ는 "불필요한 설명을 배제하고… 침묵을 활용해 상대의 압박을 기다리십시오"로 확연히 갈렸고, 실제 대사도 ESTP는 장황한 반박("증거라도 있습니까?"), ISTJ는 짧은 침묵형 답변("왜 묻는 겁니까?")으로 이어졌다.
- fair-play 불변식(무고자 알리바이, 진범 하드게이트)은 이 프롬프트와 무관하게 서버가 결정론적으로 강제하는 채널(Phase 9의 `computeBreakableHardGate`, 액터 프롬프트의 게이트 지시)에 있으므로, 전략가 판단이 성향에 따라 더 널뛰어도 fair-play 자체는 깨지지 않는다는 걸 프롬프트 주석에도 명시해뒀다.

- **Technical Rationale (죽은 매개변수를 즉시 제거하기로 한 것)**:
  - 사용자가 "발견하면 확인 후 바로 없애라"고 명시적으로 요청해, 이후 세션에서도 기본 원칙으로 적용하기로 했다(메모리에 기록). 이번 건은 "제거"가 아니라 "실제로 쓰이게 고치는" 방향으로 처리했는데, 죽은 코드였던 이유가 "애초에 안 쓸 정보라서"가 아니라 "구현이 누락돼서"였고, 마침 사용자가 그 기능 자체를 원했기 때문이다.

## Phase 16-Fix: 배포 누락 발견, 시각 표현 오류, RAG 검토 (Completed)

- **Phase 13 변경사항이 실제로는 배포되지 않았던 걸 뒤늦게 발견**: Phase 15(전략가 페르소나 반영) 배포 후 사용자가 "지금 실행하면 새 버전이냐"고 물어 확인하는 과정에서, `NotesPanel.tsx`/`gameSave.ts`/`InvestigationBoard.tsx` 재작성/`InterrogationChat.tsx` mode 라벨 제거 등 Phase 13의 핵심 UI 변경들이 로컬에 빌드·린트까지 통과한 채로 남아있을 뿐 한 번도 `git commit`/`push`된 적이 없었다는 걸 발견했다 — Phase 14·15 커밋이 성공적으로 배포된 것과 별개로, 그보다 먼저 했어야 할 Phase 13 커밋 자체를 빠뜨렸던 것. 배포된 JS 번들을 직접 grep해서("카드를 클릭하면" 같은 신규 문구가 있는지) 확인하는 방식으로 발견하고 즉시 커밋·푸시해 반영했다.
- **사인 시각 표현 오류**: `CASE_OVERVIEW.timeOfDeath`가 "워크숍 둘째 날 새벽 23:45경"으로 되어 있었는데, 23:45(밤 11시 45분)는 "새벽"(자정 이후~해뜨기 전)이 아니다. "밤 23:45경"으로 수정. (참고: `PROSE_INTRO`의 "새벽 06시경"은 시신 발견 시각이라 맞는 표현이었음 — 사인 쪽만 오류였다.)
- **RAG로 속도 개선이 가능한지 검토**: 사용자가 "AI 성향을 RAG 방식으로 만들면 빨라질까"라고 질문. RAG는 대용량 자료에서 관련 조각만 검색해 프롬프트에 넣는 기술인데, 저희 페르소나(8종, 필드 5개짜리)·캐릭터 시트는 이미 극히 작고, 이미 "관련 배역 하나만" 코드 조회로 골라 넣고 있어 RAG가 풀어주는 문제 자체가 없다고 판단했다 — 오히려 임베딩·벡터 검색이라는 새 단계가 추가돼 더 느려질 수 있다. Phase 14에서 확인한 실제 병목(Vercel 발신 경로, 턴당 전략가+액터 순차 2콜)은 "프롬프트에 뭘 담는가"가 아니라 "API를 몇 번·어디로 호출하는가"의 문제라 RAG로 손댈 수 있는 영역이 아니다. 결론: RAG 도입 없이, 여전히 유효한 유일한 절대속도 개선 레버는 2콜→1콜 통합(보류 중)뿐이다.

- **Technical Rationale (배포 누락을 어떻게 다시 안 일으킬지)**:
  - 이번 사고의 교훈: 로컬 build/lint 통과와 실제 배포 반영은 별개다. 앞으로 "완료했다"고 보고하기 전에 `git status`로 미커밋 변경이 없는지, 필요하면 배포된 번들에서 직접 grep해 실제 반영을 확인하는 절차를 습관화한다.

## Phase 17-Fix: Edge 타임아웃 실사용 재현, 전략가 콜 제거 (Completed)

- 사용자가 실제 플레이 중 "Unexpected token 'A', "An error o"... is not valid JSON" 에러를 만났다. 토큰(사용량) 문제가 아니라, 서버가 JSON 대신 HTML 에러 페이지를 반환했고 클라이언트가 그걸 JSON으로 파싱하려다 실패한 것이었다.
- **원인 확정**: Vercel 공식 문서에서 "Edge 런타임 함수는 스트리밍 없이는 응답을 25초 안에 시작해야 한다"는 조항을 확인했고, 프로덕션에 직접 curl로 3회 연속 재현한 결과 **3/3 모두 정확히 25.1~25.5초에 `FUNCTION_INVOCATION_TIMEOUT`(504)** 로 실패했다 — 우연한 지연이 아니라 사실상 항상 재현되는 상태였다. Phase 14에서 Edge 전환으로 네트워크 경로 불안정성은 해결했지만, 그 대가로 이 25초 마감시한이라는 새로운 제약이 생겼는데, 마침 전략가+액터 순차 2콜 구조의 총 소요시간이 딱 그 경계(~25초)에 걸려 있었다.
- **해결(전략가 콜 제거)**: 전략가 계층이 매 턴 판단하던 내용(붕괴조건 근접도 카운팅, 블러핑 차단)이 이미 액터 프롬프트 자체(`buildBreakdownSection`, `buildEvidenceWhitelistSection`)에 중복으로 들어있었고, 유일하게 고유했던 정보(페르소나 성향별 전략 경향, Phase 15에서 추가)는 단순 조건문이라 LLM 판단이 필요 없었다. 이 조건문을 `buildPersonaTendencySection()`으로 옮겨 액터 프롬프트에 정적으로 포함시키고, 전략가 API 콜과 `strategist-prompt.ts` 파일 자체를 완전히 제거했다. 턴당 NIM 호출이 2콜→1콜로 줄어 로컬 테스트 기준 약 2초까지 떨어졌고, ESTP/ISTJ 페르소나 차이(Phase 15에서 검증한 행동 양식 차이)도 그대로 유지됨을 재확인했다.
- **참고**: 무고자 캐릭터는 대화가 6턴을 넘으면 여전히 락아웃 판정 콜(Phase 9)이 추가로 붙어 2콜이 될 수 있다 — 다만 이 판정 콜은 체크리스트 분류만 하는 훨씬 가벼운 프롬프트라 액터 콜과 합쳐도 25초 안에 들어올 가능성이 높다고 보고 있으나, 실측 확인은 아직 안 했다(다음 단계 참고).
- **세이브/로드 UI 관련**: 사용자가 처음엔 "버튼이 없어서 기능이 빠진 줄 알았다"고 했으나, 실제로는 정상 작동 중임을 재확인(새로고침 시 이어하기 프롬프트가 뜸)했고, 버튼 없이 지금의 자동 감지 방식을 그대로 유지하기로 확정했다.
- **향후 방향으로 명시적으로 남겨둘 것 — "전략가 = 친구 대리 AI" 비전**: 사용자는 이번 제거가 순전히 속도 문제 때문의 임시 조치이고, 전략가 계층 자체는 이 프로젝트에서 중요한 요소가 될 수 있다고 판단했다. 구체적 비전: "플레이어와 함께 플레이하는 AI"를 별도로 설정해두고, 이 AI가 각 캐릭터별로 전략을 세우면 그 전략에 따라 게임 속 캐릭터가 연기하는 구조로 발전시키면, 친구들과 하는 파티 게임에서 **부족한 인원(친구) 자리를 AI가 대신 채워주는** 기능으로 확장할 수 있다는 아이디어다. 지금은 속도 제약 때문에 보류하지만, 나중에(유료 인프라 전환, NAN 측 채택으로 자체 서버 확보 등) 다시 살릴 가치가 있는 방향으로 기록해둔다.

- **Technical Rationale (전략가를 완전히 삭제가 아니라 "부활 가능한 형태"로 기록해두는 이유)**:
  - 이번 제거는 "전략가라는 개념이 틀렸다"가 아니라 "지금 이 구현(매 턴 별도 LLM 콜)이 25초 제약과 충돌한다"는 순수 인프라 문제였다. 사용자가 이 계층 자체의 장기적 가치(친구 대리 AI)를 명확히 밝혔으므로, 나중에 스트리밍 응답이나 더 넉넉한 실행시간 환경(Node/Fluid Compute, 유료 플랜, 자체 서버)으로 옮겨갈 때 그대로 되살릴 수 있도록 설계 의도를 여기 남겨둔다.

## Phase 18-Fix: Edge 런타임 재검증 실패, Node.js로 최종 복귀 (Completed)

- Phase 17의 전략가 콜 제거(2콜→1콜)를 배포한 뒤, 사용자에게 배포 확인을 부탁해 Vercel 대시보드에서 해당 배포가 **Ready/Production**임을 확인했다. 그런데도 프로덕션 curl 테스트는 여전히 정확히 ~25초에 `FUNCTION_INVOCATION_TIMEOUT`으로 실패했다 — 배포 자체는 정상이었는데 문제가 해결되지 않은 것.
- **재조사**: 같은 시점에 (1) NVIDIA API 직접 호출(1.1초), (2) 로컬 dev 서버 단일 콜(1.8초)이 둘 다 빨랐다. 즉 콜을 1개로 줄였는데도 프로덕션(Edge)에서만 단독으로 ~25초가 걸리는 모순이 관찰됐다.
- **가설**: 이전 2콜 구조가 25초 안에 "성공"했던 건 콜 수가 적어서가 아니라, 첫 번째 콜(전략가)이 Edge→NVIDIA 간의 최초 연결(TLS 핸드셰이크 등)을 힘들게 맺어두면 두 번째 콜(액터)이 그 웜 커넥션에 얹혀가 빨리 끝났기 때문일 수 있다 — 콜을 1개로 줄이면서 오히려 그 "무임승차"가 사라져, 매번 연결 수립 비용을 처음부터 다시 치르게 된 것으로 추정된다. 정확한 원인은 Vercel 대시보드 심화 로그 없이 100% 확정하긴 어려웠다.
- **결정**: Edge 런타임은 스트리밍 없이는 25초가 하드 마감이라는 게 Phase 17에서 이미 확인된 사실이었고, 콜 수를 줄여도 이 마감을 못 지키는 게 재확인됐으므로, Edge 자체를 포기하고 세 라우트(`interrogate`/`accuse`/`casting`) 전부 Node.js(Fluid Compute, 기본 300초)로 되돌렸다.
- **검증**: 배포 후 4회 연속 테스트 결과 전부 200 성공(94.7초/79.3초/48.0초/92.8초) — Phase 12/14에서 봤던 것과 같은 큰 변동성이 돌아왔지만, **적어도 매번 실제로 응답이 온다**. "느리지만 결국 성공"이 "빠른 척하다 100% 실패"보다 낫다고 판단해 이 상태로 확정했다.
- 전략가 콜 제거(Phase 17)는 Edge를 포기한 뒤에도 그대로 유지한다 — 콜 수를 줄이는 것 자체는 Node 런타임에서도 총 대기시간을 줄여주는 순수한 이득이고, 페르소나 성향 반영도 그대로 보존되기 때문이다.

- **Technical Rationale (Edge 시도 자체가 낭비는 아니었던 이유)**:
  - Edge 전환은 실패로 끝났지만, 이 과정에서 "우리 병목이 정확히 어디에 있는가"(네트워크 변동성 vs 응답시간 마감 vs 콜 수)를 하나씩 실측으로 좁혀나갔다 — Phase 12(레이트리밋 의심) → Phase 14(공유 IP 확정, Edge 채택) → Phase 17(Edge 25초 마감 발견, 콜 수 감소) → Phase 18(콜 수 감소로도 못 고침, Node 복귀)까지 매 단계 데이터로 가설을 검증하고 다음 가설로 넘어갔다. 최종 선택(Node 복귀)이 나쁜 소식이더라도, 그 과정에서 "우리가 구체적으로 무엇과 씨름하고 있는지"는 훨씬 선명해졌다.

## Phase 19-Fix: 내부판정 브라켓 노출 버그, Vercel→Cloudflare Workers 전환 (Completed)

- **내부판정 브라켓 노출(플레이어 스크린샷 제보)**: 아이패드로 플레이 중이던 사용자가 "[비밀노출: 없음]", "[심문종료: 아니오]"가 대사 위에 그대로 노출되는 스크린샷과, 같은 질문을 연달아 보냈을 때 "The string did not match the expected pattern"이라는 별개 오류가 뜨는 스크린샷을 제보했다. 원인: 모델이 가끔 `[내부판정: ...]` 래퍼로 묶지 않고 `비밀노출`/`심문종료`를 독립된 브라켓으로 따로 출력해, "내부판정" 라벨만 찾는 정규식이 놓쳤다(한 사례는 앞부분만 걷어내지고 ", 심문종료:아니오]" 꼬리만 남기도 함). 이 값들은 Phase 9 이후 실제 락아웃 판정에 전혀 쓰이지 않으므로(진범은 하드게이트, 무고자는 전용 판정콜), 값을 다시 파싱할 필요 없이 화면에서 안 보이게 걷어내기만 하면 되는 문제였다. `LEAKED_INTERNAL_FIELD_RE`를 추가해 "비밀노출/심문종료/증거카테고리/붕괴조건충족/신발요청" 라벨을 브라켓 유무·선행 쉼표와 무관하게 걷어내도록 했고, 제보받은 두 leak 문자열 그대로 유닛 테스트해 확인 후 배포했다.
- **Vercel → Cloudflare Workers 전환**: Phase 18에서 응답 속도가 다시 불안정(47~93초)해지자, 사용자가 "로컬은 항상 빨랐다 — Vercel 쪽 문제라면 같은 역할을 하는 국내/타 플랫폼으로 옮겨볼 수 있지 않냐"고 제안했다. `cloudflare-pages-experiment` 브랜치를 새로 만들어 `main`(Vercel 배포 중)은 전혀 건드리지 않은 채 실험을 진행했다:
  - `@opennextjs/cloudflare` + `wrangler`로 Cloudflare Workers 배포 지원 추가(`wrangler.jsonc`, `open-next.config.ts`, `nodejs_compat` 플래그).
  - 로컬 `opennextjs-cloudflare preview`(실제 workerd 런타임)로 캐스팅·심문(openai SDK + Buffer 기반 castingToken)이 전부 정상 동작함을 먼저 확인.
  - Cloudflare 대시보드에서 GitHub 연동 프로젝트 생성 → 처음엔 프로덕션 브랜치가 기본값 `main`으로 잡혀 빌드 실패(설정 파일이 실험 브랜치에만 있었음) → `cloudflare-pages-experiment`로 브랜치 지정 후 빌드 성공.

## Phase 21-Fix: 소지품 요청 판정 라운드-종료 배치 처리 + 라운드 전환 화면 + 증언 카드 스포일러 제거 (Completed)

Phase 20 완료 후 사용자가 "신발 외 다른 물품도 요청할 수 있게 하고 싶다"고 제안해, 세 단계에 걸쳐 설계를 다듬었다.

- **1차 시도(폐기)**: 매 심문 턴마다 `[행동판정: 물품요청=<이름>]` 구조화 신호로 즉시 판정하려 했다. 사전 등록 물품(신발)은 서버 키워드 폴백 덕에 100% 안정적이었지만, 등록 안 된 임의 물품(예: 지갑)은 대사로는 분명히 응했는데도 판정 필드가 계속 "없음"으로 나오는 사례가 반복 관측됐다(지시문을 세 차례 수정 — 조건부 생략→항상 출력, 예시 추가 — 해도 개선 안 됨, 심지어 `물품요청=X`처럼 프롬프트의 `<물품명>` 표기를 그대로 베끼는 사례까지 나옴). 무고자 락아웃 판정과 똑같은 실패 패턴(롤플레이 생성 + 자기 판정 동시 수행이 불안정)이라고 결론짓고 이 방식 자체를 포기했다.
- **2차 제안(사용자, 채택)**: "라운드가 끝날 때(다음 라운드 클릭 시) 그 라운드 대화 전체를 한 번에 검토해서 요청된 물품을 다음 라운드 조사모드에 일괄 반영하면 어떠냐"는 제안. 대사 생성과 완전히 분리된 별도 판정 콜(`buildItemRequestReviewPrompt`, `/api/round-review`)로 구현 — 락아웃 판정 콜과 동일한 원리로, 롤플레이 부담 없이 "읽고 추출만" 하니 신뢰도가 크게 올랐다(사전 등록 물품은 로컬 테스트 5/5 100%, 임의 물품도 지시문 보강 후 대략 50~75%까지 개선 — 임의 물품은 순수 flavor라 완벽하지 않아도 게임에 지장 없음).
  - 사용자가 한 걸음 더 나아가 "기존 신발 같은 진짜 증거도 같은 방식(라운드-종료 배치)으로 통일하자"고 제안했다. 처음엔 신발이 진범 붕괴 하드게이트(카테고리 B)에 쓰이는 핵심 물증이라 즉시 반영이 필요하다고 반대했으나, 사용자가 "즉시 반응하는 증거 = 진짜 증거"라는 메타 신호를 플레이어가 눈치채면 안 되고, 라운드 안에 못 찾은 증거는 실제로 오답 방향으로 가는 게 맞는 설계라고 반박 — 타당한 지적이라 판단해 전면 수용했다. `scoring.ts`의 `breakdownBonus`가 실시간 "붕괴 판정"이 아니라 최종 `revealedEvidenceIds`만으로 계산된다는 걸 먼저 확인해, 지연 반영이 최종 점수에는 영향 없음을 검증한 뒤 진행했다.
  - 결과적으로 `interrogate/route.ts`의 매 턴 소지품 판정 로직(`SHOE_EVIDENCE_BY_CHARACTER`, 키워드 폴백, `unlockedEvidenceId`/`adHocEvidence` 응답 필드)을 전부 제거했다 — 액터 프롬프트도 "어떻게 연기할지"(`buildItemComplianceSection`)만 남기고 판정 신호는 완전히 뺐다. 3라운드(마지막 라운드) 종료 후에는 이 리뷰 콜 자체를 호출하지 않는다 — 다음 라운드 조사모드가 없어 반영할 곳이 없고, "제때 조사 못하면 증거를 못 찾은 것"이라는 의도적 설계.
- **라운드 전환 화면 추가**: "화면에 큰 변화가 없어서 라운드가 바뀐 건지 애매했다"는 별도 피드백에 맞춰, 라운드가 바뀔 때마다 `RoundTransitionScreen`을 새로 넣었다. 게임 시작 시 "총 3라운드로 진행됩니다 / 라운드 1 — 주어진 정보를 바탕으로 심문하세요", 2라운드부터는 "확보한 증거품과 새로 공개된 정보를 바탕으로 심문하세요"(+ round-review로 새로 확보된 물증 목록 표시), 마지막 라운드는 "(마지막 라운드) 지금까지 수집한 증거를 바탕으로 최종 심문을 진행하세요"로 문구를 달리했다. 이 화면이 자연스럽게 위 배치 리뷰 결과를 공개하는 자리도 겸한다.

- **Technical Rationale (배치 처리로 전환한 게 근본적 해결이었던 이유)**:
  - 이 세션 초반에 사용자가 "문제가 생길 때마다 그 증상만 땜질하면 프롬프트가 계속 비대해지고 새 문제를 낳는다"고 경고했었다. 매 턴 인라인 판정을 계속 프롬프트 문구로 고치려던 첫 시도가 정확히 그 패턴이었다 — 세 번 고쳐도 안 됐다. 반면 "판정을 대사 생성에서 완전히 분리하고, 신뢰도가 필요한 작업엔 전용 콜을 쓴다"는 원칙은 이 프로젝트에서 무고자 락아웃(Phase 9)에 이미 검증된 패턴이었고, 이번에도 같은 원칙을 적용하니 실제로 더 잘 작동했다. 사용자가 제안한 "라운드 단위 배치"는 이 원칙을 한 단계 더 밀어붙인 것 — 매 턴이 아니라 라운드당 한 번이라 호출 비용도 낮고, 모델이 참고할 맥락(전체 대화)도 더 넓어 판정 품질도 같이 올라간, 양쪽 다 이기는 방향이었다.

Phase 21 배포 후 사용자가 "지갑 외에 다른 물품도 더 테스트해달라, 3라운드에 요청한 게 다음 라운드가 없어서 오류가 나지 않는지 확인해달라"고 요청했다.

- **추가 물품 테스트**: 열쇠·손목시계(응함) + 목걸이·다이어리(거부) + 신발(응함)을 한 대화에 섞은 5물품 시나리오를 여러 번 반복 테스트한 결과, 응한 물품만 정확히 걸러내고(열쇠·손목시계·신발) 거부한 물품(목걸이·다이어리)은 정확히 제외되는 걸 확인했다 — 로컬 3/3, 3/3 재현.
- **버그 발견(round-review 파서)**: 브라우저로 실제 플레이(라운드1~3, "다음 라운드" 클릭)를 재현하며 신발/지갑 요청을 넣었는데, 똑같은 대화 내용을 curl로 재현하면 항상 성공하는데 정작 실제 게임 플레이에서는 두 번 다 감지에 실패했다. 서버에 임시 디버그 로그를 심어 원인을 추적한 결과, 모델이 가끔 `[요청물품1: 신발]` 대신 여는 대괄호 없이 `요청물품1: 신발]`을 출력하는데, `round-review/route.ts`의 `REQUESTED_ITEM_RE`가 여는 대괄호(`\[`)를 필수로 요구하고 있어서 이 사례를 통째로 놓치고 있었다. 정확히 이 파일 내 다른 브라켓 필드들(`MODE_RE`, `JUDGMENT_RE`, `ACTION_RE`)이 이미 `\[?...\]?`로 대괄호 유무를 허용하도록 만들어져 있던 것과 똑같은 실패 패턴인데, 새로 만든 이 정규식만 그 관례를 놓쳤던 것 — 새 코드를 작성할 때 기존 코드베이스에 이미 검증된 방어 패턴을 놓치기 쉽다는 교훈. `\[?요청물품\d*\s*[:=]\s*([^\]\n]+)\]?`로 수정하고 디버그 로그는 제거했다. 수정 후 이전에 실패했던 정확히 그 대화 내용으로 3/3, 이전 테스트 케이스들도 전부 재검증해 모두 통과했다 — 앞서 Phase 21에서 "임의 물품은 50~75% 정도"라고 보고했던 신뢰도 추정치는 사실 이 파싱 버그 때문에 저평가된 것이었고, 실제로는 사전 등록 물품과 거의 동급으로 안정적이다.
- **3라운드 "때늦은 요청" 결과 화면 표시**: 사용자 제안 — "3라운드에 요청한 물품은 다음 라운드가 없어 반영이 안 되는데, 이걸 그냥 버리지 말고 결과 화면에서 '이건 너무 늦게 요청한 거였다'는 걸 보여주면 재미있는 학습 효과가 있지 않겠냐"는 아이디어. `GameApp.tsx`의 `advanceRound()`를 수정해 3라운드 종료 시에도 round-review를 호출하되(기존엔 아예 스킵했음), 그 결과를 `collectedEvidenceIds`에는 반영하지 않고(점수에 영향 없음 유지) `lateRoundItemNames`라는 별도 상태로 저장해 `ResultScreen.tsx`에 새 섹션으로 전달한다. 사전 등록 물증(신발 등)과 임의 물증(지갑 등) 모두 동일하게 이 섹션에 뜬다 — "제때 조사하지 않으면 놓친 것"이라는 기존 설계 원칙은 그대로 유지하면서, 그 놓침 자체를 플레이어에게 명시적으로 알려주는 피드백 레이어만 추가한 것.
- **부수 발견**: 결과 화면 재검증 중 진범 자백 장면 생성이 가끔 "저는 설정된 규칙에 따라 자백할 수 없습니다"처럼 캐릭터를 깨고 메타 발언을 하는 사례를 관측했다. 이번 세션 작업과 무관한 기존 이슈로 보이며, 범위 밖이라 이번엔 손대지 않고 다음 단계에 기록만 해둔다.
- **조사모드 라벨 축약**: "용의자 목록에 없는 다른 팀원의 증언"이 너무 길다는 피드백에 "다른 팀원의 증언"으로 줄였다(`InvestigationBoard.tsx`).

- **Technical Rationale (같은 파일 안에서도 방어 패턴이 전파되지 않을 수 있다는 교훈)**:
  - 이 프로젝트는 "모델이 브라켓을 가끔 빼먹는다"는 걸 이미 여러 번 겪고 고쳐왔다(MODE_RE 등). 그런데도 이번에 새로 만든 정규식에서 똑같은 실수를 반복했다 — 기존 코드에 이미 있는 방어 패턴이라고 해서 새 코드에 자동으로 전파되지는 않는다는 뜻이다. 이번 버그는 사용자가 "다른 물품도 더 테스트해달라"고 명시적으로 요청하지 않았다면 한동안 "임의 물품은 원래 좀 불안정하구나"로 오해한 채 넘어갔을 수 있다 — 애매한 실패를 "모델 한계"로 성급히 결론짓지 말고 실제 원인(raw 모델 출력)을 직접 찍어봐야 한다는, 이 세션 초반의 Vercel 지연 조사(Phase 12→14)와 같은 교훈이 여기서도 반복됐다.

## Phase 23-Fix: 자백 장면 메타 발언 이탈 원인 규명 및 전용 프롬프트 분리 (Completed)

Phase 22에서 백로그로만 남겨뒀던 자백 장면 메타 발언 이탈 건을 사용자가 "원인을 확인해달라"고 요청해 조사했다.

- **원인**: `accuse/route.ts`가 자백 장면을 생성할 때 심문용 `buildActorSystemPrompt`를 그대로 재사용하고 있었는데, 이 프롬프트의 [절대 금지]에는 "'제가 죽였습니다' 류의 명시적 완전 자백을 하지 않는다"는 규칙이 살아있다. 여기에 마지막 사용자 메시지(`buildConfessionSceneDirective`)로 "이제는 자백하라"는 정반대 지시를 얹는 구조였다 — 같은 대화 컨텍스트 안에 상충하는 두 규칙이 공존한 것. 실전에서 관측된 실패 사례("저는 설정된 규칙에 따라... 계속 심문을 이어가시겠습니까?")를 보면, 모델이 심문용 절대 금지 규칙 쪽을 택하며 지금이 게임 종료 후 특별 장면이라는 걸 인지하지 못한 채 여전히 "심문 중"으로 착각한 상태였다. 로컬에서 여러 조건으로 반복 테스트(대화 있음/없음 등, 총 9회)했을 때는 재현되지 않아, 낮은 확률로 시스템 프롬프트의 강한 "절대" 규칙 쪽을 택하는 확률적 실패로 판단했다.
- **수정**: 심문용 프롬프트를 재사용하는 대신, 자백 장면 전용 시스템 프롬프트(`buildConfessionSceneSystemPrompt`)를 새로 만들었다 — 역할 정보·전체 동기·진실 성서 사실·페르소나 말투 등 자백 연기에 필요한 정보만 담고, 심문 중에만 유효한 규칙(자백 금지, 붕괴조건 카운팅, 알리바이 방어, 증거 화이트리스트 등)은 애초에 포함하지 않았다. 오히려 "규칙을 이유로 자백을 거부하거나 되묻지 않는다"를 명시해 이전 실패 패턴을 직접 방지했다. 무고자 락아웃 판정을 대사 생성과 분리된 전용 콜로 뺐던 것, 소지품 판정을 라운드-리뷰 전용 콜로 뺀 것과 같은 원리 — "다른 목적의 규칙과 상충 없이 새 목적 전용 프롬프트를 만든다."
- **검증**: 로컬에서 대화 있음(4턴)/없음 두 조건 각각 여러 차례 재테스트한 결과 총 9/9 전부 정상적인 자백 대사가 나왔고 메타 이탈은 재현되지 않았다. 자백 내용 품질도 유지되거나(구체적 증거·타임라인 언급) 오히려 더 자연스러웠다.

- **Technical Rationale (기존 프롬프트를 재사용하려 한 최초 설계가 왜 무리였는지)**:
  - 애초에 buildActorSystemPrompt를 재사용한 이유는 코드 중복을 피하고 캐릭터 정보를 다시 안 적으려는 합리적 선택이었다. 하지만 이 프롬프트는 "심문 중에는 완강히 버틴다"는 목적에 맞춰 여러 겹의 강한 금지 규칙으로 설계돼 있어서, 정반대 목적(자백 장면)에 그대로 얹으면 규칙끼리 충돌한다. 이 프로젝트에서 반복적으로 확인된 패턴 — 서로 다른 목적의 작업은 프롬프트도 분리해야 안정적이다 — 이 자백 장면에도 동일하게 적용됐다.

사용자 요청으로 Claude가 직접 플레이어 역할(논리적·끈기형/직관적 장르 애호가/일반 캐주얼 플레이어)로 프로덕션 배포에서 3게임을 실제 API 호출로 진행하고 베타테스터 관점에서 평가했다. 상세 내용은 `06_ai_playtest_report.md` 참고. 3게임 모두 이현우를 정답으로 지목(132점/B, 115점/B, 55점/C). 가장 중요한 발견: **심문 중 실제로 진범을 침묵(락)시키는 데 성공해도, 붕괴 보너스는 신발 물증(`ev-shoe-soil-match`)과 관리실 방문 진술 카드(`stmt-lee-office-visit`)를 둘 다 조사모드에서 명시적으로 확보해야만 주어져서 — 3게임 중 2게임에서 "라이브 락은 성공했는데 보너스는 0점"인 괴리가 발생했다. 이 조건이 화면에 안내되지 않아 플레이어가 이해하기 어렵다. 그 외엔 경미한 오탈자(문법 오류) 몇 건 외에 특별한 문제는 재현되지 않았고, 최근 수정된 항목들(배경 인지, 직책 인지, 브라켓 파싱)은 전부 안정적으로 작동했다.

## Phase 25-Fix: "다른 팀원의 증언" 카드 내부 연출 노트 노출 버그 수정 (Completed)

사용자가 새 창을 열어 "소스 코드를 절대 읽지 말고 배포된 URL로만 블라인드 플레이하라"는 프롬프트로 다른 Claude 인스턴스에게 플레이를 맡겼다. 그 결과 리포트에서 실제 버그를 하나 짚어냈다 — 3라운드 "다른 팀원의 증언" 섹션의 카드 세 개가 실제 증언이 아니라 내부 연출 지시문/개발 메모처럼 보이는 텍스트를 그대로 노출하고 있었다.

- **원인**: Phase 21에서 "다른 팀원의 증언" 카드로 재구성할 때 카드 **제목(라벨)**만 스포일러 없이 바꾸고(`e.name` 대신 통일된 헤더), **내용(`revealedFact`)** 자체는 그대로 뒀다. 당시 확인한 `stmt-park-dispute-reason`("박서연-김영훈 성과 갈등")은 마침 깔끔한 문구였지만, 나머지 세 항목은 원래부터 내부 연출 노트 스타일로 쓰여 있었다는 걸 놓쳤다:
  - `stmt-lee-office-visit`: "관리실 방문 = CCTV 조작 시도 정황" — 붕괴 트리거 질문("관리실은 왜 갔냐")의 정답을 그대로 알려주는 문구였다. 가장 심각한 스포일러.
  - `stmt-lee-park-grudge`: "...박서연 심문 시 먼저 적개심이 새어나오고... 이현우 심문 시엔... '그건 내 성과였다'며 반박" — AI 캐릭터를 어떻게 연기시킬지 적어둔 내부 스크립트가 그대로 노출.
  - `stmt-motive-disclosure`: "인사평가 위협감을 스스로 인정하는지" — 증언이 아니라 판정 기준 설명문.
- **수정**: 세 항목의 `revealedFact`를 자연스러운 증언 문장으로 다시 썼다. `stmt-lee-park-grudge`의 배역별 반응 스크립트 부분은 애초에 `characters.ts`의 각 배역 `knownSecrets`에 이미 중복으로 들어있어서(박서연·이현우 각자의 비대칭적 인지 상태 포함) 안전하게 제거할 수 있었다 — AI 연기 지침은 그쪽에서 이미 담당하고 있으므로.
- `revealedFact`는 플레이어 카드 표시(`InvestigationBoard.tsx`) 외에 액터 프롬프트의 증거 화이트리스트·진술 게이트(`actor-prompt.ts`)에도 쓰이는데, 두 곳 다 텍스트 내용을 자연어로 바꿔도 의미(관리실 방문 사실 자체는 확정됐다는 점 등)는 그대로 전달되므로 AI 판단 로직에는 영향 없음을 확인했다. 붕괴 조건 판정(`computeBreakableHardGate`)은 애초에 `breakdownCategory` 태그·키워드 매칭만 쓰고 텍스트 내용을 안 보므로 이번 수정과 완전히 무관하다.
- 로컬 브라우저로 3라운드까지 재현해 카드 텍스트가 정상적인 증언 문장으로 보이는 것을 확인했다.

- **Technical Rationale (라벨만 고치고 내용은 안 본 게 왜 문제였는지)**:
  - Phase 21에서 "카드 제목이 스포일러다"라는 피드백을 받았을 때, 문제의 범위를 "제목"으로만 한정하고 고쳤다. 그런데 실제로 플레이어에게 노출되는 건 제목이 아니라 본문(`revealedFact`)이었고, 그 안에 훨씬 심각한 스포일러(정답 그 자체)가 들어있었다 — 다른 사람(이번엔 다른 Claude 인스턴스)이 실제로 블라인드 플레이를 해봐야만 발견되는 종류의 문제였다. 코드를 훑어보며 "괜찮아 보인다"고 판단하는 것과, 실제로 플레이어 입장에서 화면에 뜨는 걸 클릭해보는 것 사이엔 차이가 있다는 걸 보여준 사례.

## Phase 26: 프롤로그에 플레이 방법 안내 산문 추가 (Completed)

Phase 25의 블라인드 플레이테스트 리포트에서 "신발/지갑 같은 소지품을 요청해서 증거로 얻어내는 게 플레이어 몫이라는 걸 게임 어디서도 안내받지 못했다", "심문 가능 횟수(락 조건)가 있다는 걸 캐릭터가 갑자기 잠겨서야 알았다"는 두 지적이 나왔다. 사용자는 이걸 별도의 "게임 설명" 화면으로 만들기보다, 기존 `PROSE_INTRO`(프롤로그) 산문에 형사에게 주는 수사 조언처럼 자연스럽게 녹여 넣자고 제안했다.

- `PROSE_INTRO`에 마지막 문단을 추가: "필요한 건 가만히 있는다고 나오지 않는다. 소지품이든 진술이든, 의심스러운 게 있다면 당신이 직접 요구해서 받아내야 한다. 다만 그들의 인내심도 무한하지 않다 — 지나치게 몰아붙이거나 너무 늦게 손을 쓰면, 어느 순간 더 이상 입을 열지 않을 수도 있다."
- 정확한 메커니즘(붕괴 조건의 카테고리 수, 무고자의 비밀 소진 조건 등)은 스포일러가 되므로 명시하지 않고, "인내심에는 한계가 있다"는 암시만 남겨 긴장감을 유지했다.
- 별도의 "게임 설명" 모달/화면을 새로 만들지 않은 이유: 이미 Phase 20에서 배경 설정(연수원 정보)을 같은 방식(산문 인트로에 자연스럽게 통합)으로 성공적으로 처리한 전례가 있어, 같은 패턴을 유지하는 게 UI 요소를 늘리지 않으면서도 몰입감을 해치지 않는 방향이라고 판단했다.

## Phase 27: "AI 친구" 컨셉 도입 — 결과 화면 디브리핑 + 프롤로그 재구성 + 8명 이름 (Completed)

다른 세션의 블라인드 플레이 피드백(정민아 서브플롯이 결과 화면에 안 나옴, 소지품 요청/심문 한계가 안내 안 됨)을 계기로, 사용자가 근본적으로 다른 해결 방향을 제안했다: "게임이 끝나면 AI들이 캐릭터 연기를 내려놓고, 자기 본연의 성향(페르소나)으로 돌아와 게임 후기를 말하게 하자." 이 프로젝트가 원래 "여러 성향의 AI 중 3명을 뽑아 배역을 맡기고 함께 파티 게임을 하는" 컨셉이라는 걸 엔딩에서 명시적으로 살리는 방향이다.

- **8명 페르소나에 이름 부여** (`personas.ts`): 여성 4명(수아·하은·유나·채원), 남성 4명(재현·시우·민재·도윤). "F 성향=여성, T 성향=남성" 같은 고정관념이 생기지 않도록 각 성별에 T/F 심문전략을 2:2로 섞어 배정했다.
- **프롤로그 재구성** (`truth-bible.ts` PROSE_INTRO): 첫 문단에 "오늘 밤, 당신은 AI 친구들과 함께 머더 미스터리 파티에 모였다..."를 추가해 게임의 메타 설정을 처음부터 밝혔다. 실제 머더 미스터리 파티 게임도 원래 "다 같이 이거 해보자"는 전제를 깔고 시작하니, 이전의 "당신은 진짜 형사다"보다 오히려 솔직하고 자연스럽다고 판단했다 — 라운드 중 심문의 긴장감은 이 틀을 안다고 줄어들지 않는다.
- **결과 화면 디브리핑으로 전면 개편**: 기존엔 진범만 "자백" 장면을 연기하고 무고자는 `motiveFull` 한 줄만 노출했는데(그래서 정민아의 법인카드 비리 같은 곁가지가 결과 화면에 전혀 안 드러났다), 이제 3배역 모두 캐릭터 연기를 내려놓고 "이 역할을 맡았던 AI 친구"로 돌아와 뒤풀이 소감을 말한다(`buildDebriefSystemPrompt`, actor-prompt.ts). 진범이든 무고자든 "나는 이런 역할이었고 심문 중 이랬다"는 구조가 동일해 프롬프트 하나로 통일했고, 부수적으로 정민아 서브플롯 문제도 함께 해결됐다.
  - 순서는 플레이어가 지목한 배역이 첫 번째, 나머지는 이어지는 순서(`accuse/route.ts`) — 오답이었을 경우 "내가 지목한 사람의 해명 → 나머지 → 진범의 진짜 자백" 순서가 자연스러운 반전 구조를 만든다.
  - 이전 자백 전용 프롬프트와 같은 이유로 심문용 `buildActorSystemPrompt`를 재사용하지 않는다 — 심문 중 규칙이 전혀 없는 새 프롬프트라 애초에 규칙 충돌이 생길 수 없다.
  - 3배역 디브리핑 생성 콜을 `Promise.all`로 병렬 호출해 지연을 최소화했다.
- **검증**: 로컬에서 실제로 1라운드 CCTV 질문 → 3라운드까지 진행 → 오답(박서연) 지목까지 재현. 결과 화면에서 박서연(지목, 오답)→이현우(진범)→정민아 순서로 정확히 나왔고, 각 AI 친구가 실제 대화 내용("CCTV 얘기 꺼내면서 압박할 때")을 구체적으로 언급하며 자기 성향(MBTI)에 맞는 톤으로 소감을 말했다. 진범 역의 "너가 끝까지 나 안 지목해 줘서 다행이다" 같은 극적 반응도 자연스럽게 나왔다. 메타 이탈이나 브라켓 누출 없이 전부 정상 작동했다.

- **Technical Rationale (자백 전용 프롬프트를 디브리핑으로 흡수한 이유)**:
  - Phase 23에서 자백 장면을 위해 "심문용 프롬프트를 재사용하지 않고 전용 프롬프트를 쓴다"는 원칙을 세웠는데, 이번에 무고자까지 같은 원칙을 적용하면서 오히려 코드가 더 단순해졌다 — 진범 전용/무고자 전용으로 프롬프트를 나눌 필요 없이, "배정된 역할이 범인이었는가"라는 사실 하나만 파라미터로 넘기면 나머지 구조(역할 정보·동기·비밀·성향·실제 대화 기록 근거)는 완전히 동일했기 때문이다.

## Phase 27-Fix: 디브리핑 말투가 성향과 무관하게 다 똑같이 친절했던 문제 수정 (Completed)

Phase 27 배포 후 사용자가 직접 결과 화면을 읽어보고 "AI들 소감이 다 너무 친절하다 — 실제 친구들이 항상 친절하고 응원만 하는 건 아니지 않냐"고 지적했다. 실제로 세 배역의 디브리핑이 성향(MBTI)과 무관하게 다들 "고생했어!", "재밌었어!" 같은 상투적이고 따뜻한 어조로 수렴해 있었다.

- **원인**: `buildDebriefSystemPrompt`에 `mbtiType`/`pressureTolerance`/`corneredReaction` 데이터는 넘겼지만, "이 성향이 소감의 말투에 구체적으로 어떻게 드러나야 하는지"는 지시하지 않고 그냥 "친구끼리 편하게"라고만 했다. 심문 행동에는 이미 `buildPersonaTendencySection`으로 성향별 차이를 명시하고 있었는데, 디브리핑에는 같은 처리가 빠져있었다.
- **수정**: `buildDebriefToneSection(persona)`를 추가해 `interrogationStrategy`(T/F)·`pressureTolerance` 조합으로 말투를 명시적으로 지시한다 — T 성향은 감정보다 사실·전략 위주로 담백하게, 압박내성이 높으면 덤덤하고 호들갑 자제. 그리고 "모든 친구가 다 따뜻하고 응원하는 태도일 필요는 없다 — 무덤덤하거나, 장난스럽게 놀리거나, 은근히 승부욕을 보이거나, 투덜대는 태도도 자연스럽다"와 "재밌었어/고생했어 같은 상투적 마무리를 매번 반복하지 않는다"를 명시했다.
- **검증**: 같은 캐릭터(이현우)에 ISTJ/ENFP/ENTJ 세 페르소나를 각각 배정해 디브리핑을 비교한 결과, ISTJ·ENTJ(T+압박내성 높음)는 건조하고 담백하게, 심지어 "다음엔 좀 더 꼼꼼하게 준비해와, 나도 안 봐줄 거니까" 같은 경쟁적/도발적인 마무리까지 나온 반면, ENFP(F+압박내성 낮음)는 여전히 드라마틱하고 감정적인 톤을 유지했다 — 성향에 따른 실제 차이가 확인됐다.
